## What Is an Action?

An **action** is a communication pattern for long-running tasks. Like a service, it has a request–response structure. Unlike a service:

| | Service | Action |
|---|---|---|
| Duration | Returns immediately | Takes time; runs until done or cancelled |
| Feedback | None | Streams progress updates while running |
| Cancellation | Not supported | Can be cancelled mid-flight |
| Use when | Short, atomic operations | Navigation, rotation, motion sequences |

An action has three parts:

```
client ──► goal ──────────────────► server
client ◄── feedback (repeated) ◄── server (while running)
client ◄── result ◄────────────── server (when done)
```

## Pattern 1 — Sequential (Before Spin Starts)

The simplest action client pattern uses `send_goal_async()` and waits for each step with `spin_until_future_complete`. This drives a temporary executor to process callbacks until the future resolves.

```python
# 1. Send goal — returns a future for goal acceptance
send_future = client.send_goal_async(goal_msg, feedback_callback=feedback_cb)
rclpy.spin_until_future_complete(node, send_future)

goal_handle = send_future.result()

# 2. Request result — returns a future for the result
result_future = goal_handle.get_result_async()
rclpy.spin_until_future_complete(node, result_future)

result = result_future.result().result
```

**This only works before `rclpy.spin()` starts.** Once the node is spinning, the executor is already running — calling `spin_until_future_complete` from inside a callback would ask it to drive itself, which deadlocks. The embedded pattern (Pattern 2) handles that case.

```python
import math

import rclpy
from rclpy.node import Node
from rclpy.action import ActionClient
from turtlesim.action import RotateAbsolute


class RotateTurtle(Node):
    def __init__(self):
        super().__init__('rotate_turtle')
        self._client = ActionClient(self, RotateAbsolute, 'turtle1/rotate_absolute')

    def rotate_to(self, theta):
        self._client.wait_for_server()

        goal = RotateAbsolute.Goal()
        goal.theta = theta

        self.get_logger().info(f'Sending goal: theta = {theta:.2f} rad')
        send_future = self._client.send_goal_async(
            goal,
            feedback_callback=self._feedback_callback,
        )
        rclpy.spin_until_future_complete(self, send_future)

        goal_handle = send_future.result()
        if not goal_handle.accepted:
            self.get_logger().error('Goal rejected')
            return

        self.get_logger().info('Goal accepted — waiting for result')
        result_future = goal_handle.get_result_async()
        rclpy.spin_until_future_complete(self, result_future)

        delta = result_future.result().result.delta
        self.get_logger().info(f'Done — rotated {delta:.2f} rad')

    def _feedback_callback(self, feedback_msg):
        remaining = feedback_msg.feedback.remaining
        self.get_logger().info(f'  remaining: {remaining:.2f} rad')


def main(args=None):
    rclpy.init(args=args)
    node = RotateTurtle()
    node.rotate_to(math.pi / 2)
    node.destroy_node()
    rclpy.shutdown()


if __name__ == '__main__':
    main()
```

## Pattern 2 — Embedded in a Running Node

When an action client lives inside a node that is already spinning, the sequential pattern deadlocks.

### Why It Hangs

An `ActionClient` registers its own internal callbacks with the node's executor to receive the goal response, feedback, and result from the server. By default, those callbacks go into the node's default `MutuallyExclusiveCallbackGroup` — the same group as your timers, subscriptions, and service callbacks.

When you call `send_goal_async()` from inside a service or timer callback and then try to wait:

```
MutuallyExclusiveCallbackGroup:
  ├── snap_heading_callback  ← currently running (called send_goal_async)
  └── ActionClient response  ← BLOCKED (same group; can't run while ↑ is running)
```

The group only allows one callback at a time. Your callback is still running while it waits — so the action client's response callback can never fire, the future never resolves, and the node hangs.

### The Fix

1. Create the `ActionClient` with its own `ReentrantCallbackGroup`. This group allows its callbacks to run concurrently with other groups.
2. Use `MultiThreadedExecutor` so separate threads can run callbacks from different groups simultaneously.
3. Replace the blocking wait with **callback chaining** — attach a `done_callback` to each future instead of waiting.

```python
from rclpy.action import ActionClient
from rclpy.callback_groups import ReentrantCallbackGroup
from rclpy.executors import MultiThreadedExecutor

# In __init__:
self._action_group = ReentrantCallbackGroup()
self._client = ActionClient(
    self,
    RotateAbsolute,
    'turtle1/rotate_absolute',
    callback_group=self._action_group,
)

# Service callback — fires send_goal_async and chains, then returns immediately:
def my_service_callback(self, request, response):
    goal = RotateAbsolute.Goal()
    goal.theta = 0.0
    future = self._client.send_goal_async(goal)
    future.add_done_callback(self._goal_response_cb)   # chain, don't wait
    response.success = True
    response.message = 'rotating...'
    return response                                      # returns immediately

def _goal_response_cb(self, future):
    goal_handle = future.result()
    if not goal_handle.accepted:
        return
    goal_handle.get_result_async().add_done_callback(self._result_cb)

def _result_cb(self, future):
    delta = future.result().result.delta
    self.get_logger().info(f'Done — delta: {delta:.2f} rad')

# In main():
executor = MultiThreadedExecutor()
executor.add_node(node)
executor.spin()
```

With `ReentrantCallbackGroup` on the client, the action client's response callbacks can run on a separate thread while the service callback has already returned. The service returns immediately; the rotation completes asynchronously, logged when done.

### The complete modified file

```python
import math
import random

import rclpy
from rclpy.node import Node
from rclpy.action import ActionClient
from rclpy.callback_groups import ReentrantCallbackGroup
from rclpy.executors import MultiThreadedExecutor
from geometry_msgs.msg import Twist
from rcl_interfaces.msg import SetParametersResult
from std_srvs.srv import SetBool
from turtlesim.action import RotateAbsolute
from turtlesim.srv import SetPen, TeleportAbsolute

COLORS = [(255, 0, 0), (0, 200, 0), (0, 0, 255)]


class CirclePublisher(Node):
    def __init__(self):
        super().__init__('circle_publisher')
        self.declare_parameter('radius', 1.0)
        self.declare_parameter('angular_speed', 1.0)
        self.pub = self.create_publisher(Twist, 'turtle1/cmd_vel', 10)
        self.pen_client = self.create_client(SetPen, 'turtle1/set_pen')
        self.teleport_client = self.create_client(TeleportAbsolute, 'turtle1/teleport_absolute')
        self.tick_count = 0
        self.revolution = 0
        self.create_timer(0.1, self.timer_callback)
        self.add_on_set_parameters_callback(self.validate_params)
        self.create_service(SetBool, 'thick_pen', self.thick_pen_callback)
        self._action_group = ReentrantCallbackGroup()
        self._rotate_client = ActionClient(
            self,
            RotateAbsolute,
            'turtle1/rotate_absolute',
            callback_group=self._action_group,
        )
        self.create_service(SetBool, 'snap_heading', self.snap_heading_callback)

    def snap_heading_callback(self, request, response):
        theta = 0.0 if request.data else math.pi
        goal = RotateAbsolute.Goal()
        goal.theta = theta
        future = self._rotate_client.send_goal_async(goal)
        future.add_done_callback(self._snap_goal_response)
        response.success = True
        response.message = f'rotating to {theta:.2f} rad'
        return response

    def _snap_goal_response(self, future):
        goal_handle = future.result()
        if not goal_handle.accepted:
            self.get_logger().warn('Snap goal rejected')
            return
        goal_handle.get_result_async().add_done_callback(self._snap_result)

    def _snap_result(self, future):
        delta = future.result().result.delta
        self.get_logger().info(f'Snap complete — rotated {delta:.2f} rad')

    def thick_pen_callback(self, request, response):
        req = SetPen.Request()
        req.width = 5 if request.data else 1
        self.pen_client.call_async(req)
        response.success = True
        response.message = f'pen width: {req.width}'
        return response

    def configure_pen(self, r, g, b, width):
        self.pen_client.wait_for_service()
        req = SetPen.Request()
        req.r = r
        req.g = g
        req.b = b
        req.width = width
        req.off = 0
        future = self.pen_client.call_async(req)
        rclpy.spin_until_future_complete(self, future)

    def timer_callback(self):
        radius = self.get_parameter('radius').value
        w = self.get_parameter('angular_speed').value
        msg = Twist()
        msg.linear.x = radius * w
        msg.angular.z = w
        self.pub.publish(msg)

        self.tick_count += 1

        ticks_per_revolution = int(2 * math.pi / (w * 0.1))
        if self.tick_count % ticks_per_revolution == 0:
            self.revolution += 1
            r, g, b = COLORS[self.revolution % len(COLORS)]
            req = SetPen.Request()
            req.r, req.g, req.b, req.width, req.off = r, g, b, 3, 0
            self.pen_client.call_async(req)

            treq = TeleportAbsolute.Request()
            treq.x = random.uniform(0.0, 11.88)
            treq.y = random.uniform(0.0, 11.88)
            treq.theta = 0.0
            self.teleport_client.call_async(treq)

    def validate_params(self, params):
        for p in params:
            if p.name in ('radius', 'angular_speed') and p.value <= 0:
                return SetParametersResult(
                    successful=False,
                    reason=f'{p.name} must be positive'
                )
        return SetParametersResult(successful=True)


def main(args=None):
    rclpy.init(args=args)
    node = CirclePublisher()
    node.configure_pen(r=255, g=0, b=0, width=3)
    node.teleport_client.wait_for_service()
    executor = MultiThreadedExecutor()
    executor.add_node(node)
    try:
        executor.spin()
    except KeyboardInterrupt:
        pass
    finally:
        node.destroy_node()
        if rclpy.ok():
            rclpy.shutdown()


if __name__ == '__main__':
    main()
```


Two state flags coordinate the snap:

- `orientation_snapped` — set to `True` the moment the snap fires, preventing re-triggering on subsequent timer ticks while the turtle remains at the goal. Reset to `False` in `goal_callback` so a new goal re-arms it.
- `rotating` — set to `True` while the action is in flight; `control_loop` publishes zero velocity and returns early until `_snap_result` clears it.

`validate_params` guards only `Kv`, `Kw`, and `tolerance` for positivity — `orientation` accepts any float (including 0 and negative angles).

The action fires from `control_loop` (a timer callback). `_snap_to_orientation` calls `send_goal_async` and returns immediately; `control_loop` returns normally afterward. The three-step chain (`_snap_to_orientation` → `_snap_goal_response` → `_snap_result`) runs asynchronously on the thread serving `_action_group`. The `ReentrantCallbackGroup` on the action client is what makes that possible — without it, the action client's internal response callbacks would share the timer's `MutuallyExclusiveCallbackGroup` and could be blocked, preventing the goal response from ever arriving.

## Summary

| Concept | Key point |
|---|---|
| Action vs service | Actions are for long-running tasks; they stream feedback and support cancellation |
| `send_goal_async(goal, feedback_callback=cb)` | Sends the goal; returns a future for goal acceptance (not the result) |
| `goal_handle.get_result_async()` | Returns a future for the final result |
| `spin_until_future_complete` (Pattern 1) | Valid only before `rclpy.spin()` — drives a temporary executor |
| Why embedded clients hang | `ActionClient` response callbacks share the caller's `MutuallyExclusiveCallbackGroup`; the caller blocks them |
| `ReentrantCallbackGroup` on `ActionClient` | Puts the client's callbacks in a separate group that can run concurrently |
| `MultiThreadedExecutor` | Provides the extra threads needed to run multiple callback groups simultaneously |
| Callback chaining | `future.add_done_callback(fn)` — fire and chain without blocking; the only valid pattern inside callbacks |
