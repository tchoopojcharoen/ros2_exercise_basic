## The Blocking Problem on the Server Side

In 3.6 you saw that an action client embedded inside a spinning node needs `ReentrantCallbackGroup` so its response callbacks are not blocked by the calling callback. Action servers have a symmetric but more severe version of the same problem.

An action server's execute callback must run a loop and not return until the goal is complete. With a `SingleThreadedExecutor`, that loop occupies the only thread — no other callback can fire until the loop exits:

```
SingleThreadedExecutor (one thread):

  ┌──────────────────────────────────────────────────────┐
  │  execute_callback — running                          │
  │  ↻  while True:                                      │
  │        completed = self.revolution - start  ← stale! │
  │        if completed >= count: ...                    │
  │        time.sleep(0.1)                               │
  │                                                      │
  │  timer_callback — CAN NEVER FIRE                     │
  │  (executor is occupied running execute_callback)     │
  └──────────────────────────────────────────────────────┘
→ self.revolution never increments → loop never exits
```

`circle_publisher` already uses `MultiThreadedExecutor` from 3.6. Placing the `ActionServer` in a `ReentrantCallbackGroup` (separate from the timer's default group) lets both run concurrently:

```
MultiThreadedExecutor:

  Thread 1 — ReentrantCallbackGroup
  ┌──────────────────────────────────────────────────────┐
  │  execute_callback (blocking while loop)              │
  │  ↻  while True:                                      │
  │        completed = self.revolution - start   ← ✓     │
  │        if completed >= count: succeed()              │
  │        time.sleep(0.1)                               │
  └──────────────────────────────────────────────────────┘

  Thread 2 — default MutuallyExclusiveCallbackGroup
  ┌──────────────────────────────────────┐
  │  timer_callback fires every 0.1 s ✓ │
  │  self.revolution increments freely   │
  └──────────────────────────────────────┘
```

## Worked Example — Adding `draw_circles` to `circle_publisher.py`

The `draw_circles` action lets a client request that the turtle complete exactly N full revolutions. The server tracks progress by monitoring `self.revolution` — an integer that `timer_callback` already increments at the end of each lap. When a goal arrives, the execute callback records the current revolution count as `start`, then loops until `self.revolution - start` reaches the requested `count`, publishing a feedback message after each poll. If the client cancels mid-way, the loop exits early and reports how many revolutions were completed. Because the execute callback blocks in a while loop, it must run on a separate thread from `timer_callback` — otherwise `timer_callback` would never fire, `self.revolution` would never increment, and the loop would never exit. `circle_publisher` already uses `MultiThreadedExecutor` from 3.6, so placing the `ActionServer` in the existing `ReentrantCallbackGroup` is all that is needed.

### Change 1 — Add imports

```python

from turtlesim_interfaces.action import DrawCircles
```

### Change 2 — Register the server in `__init__`

The `_action_group` is already a `ReentrantCallbackGroup` from 3.6. Reuse it for the server:

```python
self._draw_server = ActionServer(
    self,
    DrawCircles,
    'draw_circles',
    self._execute_callback,
    callback_group=self._action_group,
)
```

### Change 3 — Add the execute callback

```python
def _execute_callback(self, goal_handle):
    count = goal_handle.request.count
    start = self.revolution
    self.get_logger().info(f'Drawing {count} circles')

    feedback = DrawCircles.Feedback()
    result = DrawCircles.Result()

    while rclpy.ok():
        if goal_handle.is_cancel_requested:
            goal_handle.canceled()
            result.success = False
            result.revolutions_completed = self.revolution - start
            return result

        completed = self.revolution - start
        feedback.revolutions_completed = completed
        goal_handle.publish_feedback(feedback)

        if completed >= count:
            goal_handle.succeed()
            result.success = True
            result.revolutions_completed = completed
            return result

        time.sleep(0.1)

    result.success = False
    result.revolutions_completed = self.revolution - start
    return result
```

`start` captures `self.revolution` at the moment the goal is accepted. The loop reads `self.revolution` continuously — a value that `timer_callback` updates on Thread 2. When `completed >= count`, the loop returns a successful result.

## Answer Key

```python
import math
import time

import rclpy
from rclpy.node import Node
from rclpy.action import ActionServer, GoalResponse
from rclpy.callback_groups import ReentrantCallbackGroup
from rclpy.executors import MultiThreadedExecutor
from geometry_msgs.msg import Twist
from turtlesim.msg import Pose
from turtlesim_interfaces.action import GoToGoal
from turtlesim_controller.control_law import compute_go_to_goal_control


class GoToGoalActionServer(Node):
    def __init__(self):
        super().__init__('goto_goal_action_server')
        self.current_pose = None
        self._busy = False
        self.declare_parameter('Kv', 1.0)
        self.declare_parameter('Kw', 5.0)
        self.declare_parameter('tolerance', 0.1)
        self.pub = self.create_publisher(Twist, 'turtle1/cmd_vel', 10)
        self.create_subscription(Pose, 'turtle1/pose', self.pose_callback, 10)
        self._action_server = ActionServer(
            self,
            GoToGoal,
            'go_to_goal',
            self._execute_callback,
            goal_callback=self._goal_callback,
            callback_group=ReentrantCallbackGroup(),
        )

    def _goal_callback(self, goal_request):
        if self._busy:
            self.get_logger().warn('Goal rejected: server busy')
            return GoalResponse.REJECT
        return GoalResponse.ACCEPT

    def pose_callback(self, msg):
        self.current_pose = [msg.x, msg.y, msg.theta]

    def _execute_callback(self, goal_handle):
        self._busy = True
        try:
            goal = [goal_handle.request.x, goal_handle.request.y]
            self.get_logger().info(f'Executing: x={goal[0]:.2f}, y={goal[1]:.2f}')

            feedback = GoToGoal.Feedback()
            result = GoToGoal.Result()

            while rclpy.ok():
                if goal_handle.is_cancel_requested:
                    self.get_logger().info('Goal cancelled')
                    goal_handle.canceled()
                    result.reached = False
                    result.distance_remaining = self._distance_to(goal)
                    return result

                if self.current_pose is None:
                    time.sleep(0.1)
                    continue

                Kv = self.get_parameter('Kv').value
                Kw = self.get_parameter('Kw').value
                tolerance = self.get_parameter('tolerance').value

                pose = self.current_pose
                v, w, d = compute_go_to_goal_control(pose, goal, Kv=Kv, Kw=Kw)

                dx = goal[0] - pose[0]
                dy = goal[1] - pose[1]
                desired = math.atan2(dy, dx)
                heading_error = math.atan2(
                    math.sin(desired - pose[2]),
                    math.cos(desired - pose[2])
                )

                feedback.distance_remaining = d
                feedback.heading_error = heading_error
                goal_handle.publish_feedback(feedback)

                if d < tolerance:
                    self.pub.publish(Twist())
                    self.get_logger().info(f'Goal reached. Final distance: {d:.3f}')
                    goal_handle.succeed()
                    result.reached = True
                    result.distance_remaining = d
                    return result

                cmd = Twist()
                cmd.linear.x = v
                cmd.angular.z = w
                self.pub.publish(cmd)

                time.sleep(0.1)

            result.reached = False
            result.distance_remaining = 0.0
            return result
        finally:
            self._busy = False

    def _distance_to(self, goal):
        if self.current_pose is None:
            return 0.0
        dx = goal[0] - self.current_pose[0]
        dy = goal[1] - self.current_pose[1]
        return math.sqrt(dx ** 2 + dy ** 2)


def main(args=None):
    rclpy.init(args=args)
    node = GoToGoalActionServer()
    executor = MultiThreadedExecutor()
    executor.add_node(node)
    try:
        executor.spin()
    except KeyboardInterrupt:
        pass
    finally:
        node.destroy_node()
        if rclpy.ok():
            rclpy.shutdown()


if __name__ == '__main__':
    main()
```

`pose_callback` runs in the default `MutuallyExclusiveCallbackGroup` on its own thread (via `MultiThreadedExecutor`), so `self.current_pose` updates freely while the execute loop is blocking. The `current_pose is None` guard at the top of the loop handles the brief window at startup before the first pose message arrives.

`_goal_callback` is called before execution starts. `GoalResponse.REJECT` declines the goal without queuing it. `self._busy` is set at the top of `_execute_callback` (not in `_goal_callback`) to avoid a race between acceptance and the flag being raised. `try/finally` guarantees `_busy` resets whether the loop exits via success, cancellation, or shutdown.

`_distance_to` is a helper that computes the remaining Euclidean distance for the cancellation result. It returns 0.0 defensively if pose has not yet been received.

## Summary

| Concept | Key point |
|---|---|
| `.action` file | Three sections: goal, result, feedback — separated by `---` |
| ament_cmake interfaces package | Required to define and generate custom message/action types |
| `rosidl_generate_interfaces` | CMake macro that generates Python bindings from `.action` files |
| Execute callback blocks | A while loop that never returns — kills other callbacks without MT |
| Why `timer_callback` must keep firing | `self.revolution` only updates in the timer; the execute loop reads it to detect completion |
| `ReentrantCallbackGroup` on server | Execute callback runs on its own thread; timer fires freely on the default group thread |
| `goal_handle.is_cancel_requested` | Polled inside the loop — not raised as an exception |
| `goal_handle.succeed()` / `.canceled()` | Must be called before returning from execute; sets the final goal state |
| `publish_feedback(msg)` | Takes a `Feedback()` message object; call repeatedly while the loop runs |
| `GoalResponse.ACCEPT` / `.REJECT` | Returned from `goal_callback` to control whether incoming goals are accepted |
| `try/finally` around execute body | Guarantees cleanup (`_busy = False`) regardless of how the loop exits |

The next section covers QoS profiles — the policies that control how messages are delivered over topics. A mismatched QoS between publisher and subscriber causes silent message loss, which is easy to trigger with the turtlesim pose topic.
