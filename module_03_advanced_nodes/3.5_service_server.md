## Services: The Server Side

In 3.4 you wrote the *client* side of a service: your node called a service that someone else provided. This section covers the *server* side: your node provides a service that other processes can call.

The two sides are symmetric in definition but asymmetric in execution:

| | Client | Server |
|---|---|---|
| Registration | `create_client(Type, 'name')` | `create_service(Type, 'name', callback)` |
| What happens at call time | Sends a request; waits or uses a future | Callback fires automatically |
| Blocking | `call_async` + optional wait | Never blocks — populates and returns a response |

### Callback Signature

A service server callback takes two arguments — the request and a pre-built response object — and must return the response:

```python
def my_callback(self, request, response):
    # read from request fields
    # write to response fields
    return response
```

The response object is already constructed when your callback is called. You do not create it — you populate it and return it.

### Choosing a Service Type

`std_srvs` provides three general-purpose service types that cover most use cases before you need a custom interface:

| Type | Request fields | Response fields | Use when |
|---|---|---|---|
| `std_srvs/srv/Empty` | — | — | Trigger-only; no data exchanged |
| `std_srvs/srv/Trigger` | — | `bool success` `string message` | Trigger with a status reply |
| `std_srvs/srv/SetBool` | `bool data` | `bool success` `string message` | Binary toggle with a status reply |

Inspect any type with `ros2 interface show std_srvs/srv/SetBool`.

### The Deadlock Rule

A server callback runs inside the executor's spin loop. `spin_until_future_complete` also drives the executor. Calling it from inside a callback asks the executor to drive itself — it is already busy in your callback, so it cannot process the future, and execution freezes.

**Inside any callback, outgoing service calls must use `call_async` alone.** The executor will process the response after your callback returns.

```python
def my_callback(self, request, response):
    req = SomeService.Request()
    self.some_client.call_async(req)   # valid — fire and forget
    response.success = True
    response.message = 'done'
    return response
```

This is the same rule from 3.4: `call_async` alone is valid inside callbacks; `spin_until_future_complete` is only valid before `rclpy.spin()` starts.

## Worked Example — Adding `/thick_pen` to `circle_publisher.py`

The `circle_publisher` from 3.4 draws colored circles at a fixed pen width. This example adds a `/thick_pen` service that lets an external caller switch between a thick and thin trail at runtime.

Inspect the type first:

```bash
ros2 interface show std_srvs/srv/SetBool
```

Output:
```
bool data
---
bool success
string message
```

`data` is the input. `success` and `message` are the outputs.

### Change 1 — Add the import

```python
from std_srvs.srv import SetBool
```

### Change 2 — Register the server in `__init__`

```python
self.create_service(SetBool, 'thick_pen', self.thick_pen_callback)
```

### Change 3 — Add the callback

```python
def thick_pen_callback(self, request, response):
    req = SetPen.Request()
    req.width = 5 if request.data else 1
    self.pen_client.call_async(req)
    response.success = True
    response.message = f'pen width: {req.width}'
    return response
```

`request.data` is read to choose a width. `call_async` dispatches the SetPen call without blocking. `response.success` and `response.message` are written before returning.

### The complete modified file

```python
import math
import random

import rclpy
from rclpy.node import Node
from geometry_msgs.msg import Twist
from rcl_interfaces.msg import SetParametersResult
from std_srvs.srv import SetBool
from turtlesim.srv import SetPen, TeleportAbsolute

COLORS = [(255, 0, 0), (0, 200, 0), (0, 0, 255)]


class CirclePublisher(Node):
    def __init__(self):
        super().__init__('circle_publisher')
        self.declare_parameter('radius', 1.0)
        self.declare_parameter('angular_speed', 1.0)
        self.pub = self.create_publisher(Twist, 'turtle1/cmd_vel', 10)
        self.pen_client = self.create_client(SetPen, 'turtle1/set_pen')
        self.teleport_client = self.create_client(TeleportAbsolute, 'turtle1/teleport_absolute')
        self.tick_count = 0
        self.revolution = 0
        self.create_timer(0.1, self.timer_callback)
        self.add_on_set_parameters_callback(self.validate_params)
        self.create_service(SetBool, 'thick_pen', self.thick_pen_callback)

    def thick_pen_callback(self, request, response):
        req = SetPen.Request()
        req.width = 5 if request.data else 1
        self.pen_client.call_async(req)
        response.success = True
        response.message = f'pen width: {req.width}'
        return response

    def configure_pen(self, r, g, b, width):
        self.pen_client.wait_for_service()
        req = SetPen.Request()
        req.r = r
        req.g = g
        req.b = b
        req.width = width
        req.off = 0
        future = self.pen_client.call_async(req)
        rclpy.spin_until_future_complete(self, future)

    def timer_callback(self):
        radius = self.get_parameter('radius').value
        w = self.get_parameter('angular_speed').value
        msg = Twist()
        msg.linear.x = radius * w
        msg.angular.z = w
        self.pub.publish(msg)

        self.tick_count += 1

        ticks_per_revolution = int(2 * math.pi / (w * 0.1))
        if self.tick_count % ticks_per_revolution == 0:
            self.revolution += 1
            r, g, b = COLORS[self.revolution % len(COLORS)]
            req = SetPen.Request()
            req.r, req.g, req.b, req.width, req.off = r, g, b, 3, 0
            self.pen_client.call_async(req)

            treq = TeleportAbsolute.Request()
            treq.x = random.uniform(0.0, 11.88)
            treq.y = random.uniform(0.0, 11.88)
            treq.theta = 0.0
            self.teleport_client.call_async(treq)

    def validate_params(self, params):
        for p in params:
            if p.name in ('radius', 'angular_speed') and p.value <= 0:
                return SetParametersResult(
                    successful=False,
                    reason=f'{p.name} must be positive'
                )
        return SetParametersResult(successful=True)


def main(args=None):
    rclpy.init(args=args)
    node = CirclePublisher()
    node.configure_pen(r=255, g=0, b=0, width=3)
    node.teleport_client.wait_for_service()
    try:
        rclpy.spin(node)
    except KeyboardInterrupt:
        pass
    finally:
        node.destroy_node()
        if rclpy.ok():
            rclpy.shutdown()


if __name__ == '__main__':
    main()
```

No new entry point. No `package.xml` change — `std_srvs` is part of the ROS2 base install.

## Answer Key

```python
import argparse

import rclpy
from rclpy.node import Node
from geometry_msgs.msg import Twist, Point
from turtlesim.msg import Pose
from rcl_interfaces.msg import SetParametersResult
from std_srvs.srv import SetBool
from turtlesim_controller.control_law import compute_go_to_goal_control


class TurtleController(Node):
    def __init__(self, frequency=10.0):
        super().__init__('turtle_controller')
        self.current_pose = None
        self.current_goal = None
        self.enabled = True
        self.declare_parameter('Kv', 1.0)
        self.declare_parameter('Kw', 5.0)
        self.declare_parameter('tolerance', 0.1)
        self.pub = self.create_publisher(Twist, 'turtle1/cmd_vel', 10)
        self.create_subscription(Pose, 'turtle1/pose', self.pose_callback, 10)
        self.create_subscription(Point, 'goal', self.goal_callback, 10)
        self.create_timer(1.0 / frequency, self.control_loop)
        self.add_on_set_parameters_callback(self.validate_params)
        self.create_service(SetBool, 'controller_enabled', self.controller_enabled_callback)
        self.get_logger().info(f'Control loop frequency: {frequency} Hz')

    def controller_enabled_callback(self, request, response):
        self.enabled = request.data
        response.success = True
        response.message = 'controller enabled' if request.data else 'controller disabled'
        return response

    def pose_callback(self, msg):
        self.current_pose = [msg.x, msg.y, msg.theta]

    def goal_callback(self, msg):
        self.current_goal = [msg.x, msg.y]

    def control_loop(self):
        if not self.enabled:
            self.pub.publish(Twist())
            return
        if self.current_pose is None or self.current_goal is None:
            return
        Kv = self.get_parameter('Kv').value
        Kw = self.get_parameter('Kw').value
        tolerance = self.get_parameter('tolerance').value
        v, w, d = compute_go_to_goal_control(
            self.current_pose, self.current_goal, Kv=Kv, Kw=Kw
        )
        cmd = Twist()
        if d < tolerance:
            self.pub.publish(cmd)
            return
        cmd.linear.x = v
        cmd.angular.z = w
        self.pub.publish(cmd)

    def validate_params(self, params):
        for p in params:
            if p.value <= 0:
                return SetParametersResult(
                    successful=False,
                    reason=f'{p.name} must be positive'
                )
        return SetParametersResult(successful=True)


def main():
    parser = argparse.ArgumentParser()
    parser.add_argument('--frequency', type=float, default=10.0)
    args, remaining = parser.parse_known_args()

    rclpy.init(args=remaining)
    node = TurtleController(frequency=args.frequency)
    try:
        rclpy.spin(node)
    except KeyboardInterrupt:
        pass
    finally:
        node.destroy_node()
        if rclpy.ok():
            rclpy.shutdown()


if __name__ == '__main__':
    main()
```

`controller_enabled_callback` sets `self.enabled` then writes both response fields and returns. No `call_async` here — `self.pub` is a publisher owned by this node, not an external service, so publishing directly is always safe. In `control_loop`, when disabled, a zero `Twist()` is published so the turtle comes to a definite stop on every timer tick rather than coasting.

Compare this with `thick_pen_callback`: the callback shape is identical (read request, do work, write response, return), but the "do work" step is different — one calls an external service via `call_async`, the other modifies a flag and publishes directly.

## Summary

| Concept | Key point |
|---|---|
| `create_service(Type, 'name', cb)` | Registers the server; `cb` fires automatically when a client calls |
| Callback signature | `def cb(self, request, response): ... return response` |
| Response object | Pre-built by the executor — populate fields and return it |
| `std_srvs/srv/Empty` | No fields on either side — use for pure triggers |
| `std_srvs/srv/Trigger` | Response only: `bool success`, `string message` |
| `std_srvs/srv/SetBool` | Request: `bool data`. Response: `bool success`, `string message` |
| `call_async` inside a callback | Valid — executor processes the response after callback returns |
| `spin_until_future_complete` inside a callback | Deadlock — executor is already running your callback |
| Direct publish inside a callback | Always valid — publishers are not services, no future involved |
