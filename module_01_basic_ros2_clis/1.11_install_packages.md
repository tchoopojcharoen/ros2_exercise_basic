## Binary Packages and the apt Repository

Every ROS2 package you have used so far — `turtlesim`, `geometry_msgs`, `rqt_graph` — was installed as a **binary package**: a pre-compiled distribution that landed on your system when you installed `ros-<distro>-desktop`.

Binary packages come from the official ROS2 apt repository at `packages.ros.org`. The ROS2 team and package maintainers build them on a schedule tied to each ROS2 release. When you install one, no compilation happens on your machine — you get ready-to-run executables directly.

This is different from **source packages**: code you write yourself (or clone from a repository) and build locally with `colcon`. Module 2 covers source packages. The two approaches are complementary:

| | Binary (apt) | Source (colcon) |
|---|---|---|
| **What** | Pre-compiled community packages | Your own code or unreleased packages |
| **Install** | `sudo apt install` | `colcon build` in your workspace |
| **Rebuild needed?** | No | Yes, after every change |
| **When to use** | Stable, released packages | Development, custom nodes |

### Search Before You Build

Before implementing any functionality from scratch, search the official ROS2 package index at **https://index.ros.org/**. The index lists every released ROS2 package across all distributions, with links to source code and documentation. There is a good chance that a well-tested, community-maintained package already does what you need — saving you days of development and debugging.

If a package appears in the index and supports your distro, its apt name follows the naming convention below and it can be installed immediately.

### Naming Convention

Every ROS2 package in the apt repository follows the same naming rule:

```
ros-<distro>-<package-name>
```

Underscores in the ROS2 package name become hyphens in the apt name. For example, on Humble:

| ROS2 package name | apt package name |
|---|---|
| `turtlesim` | `ros-humble-turtlesim` |
| `image_publisher` | `ros-humble-image-publisher` |
| `rqt_graph` | `ros-humble-rqt-graph` |

Knowing this rule lets you derive the apt name for any distro: replace `humble` with your distro name.

## About image_publisher

`image_publisher` is a ROS2 package that contains a single node: `image_publisher_node`. Given an image file on disk, it reads that file and publishes its contents continuously as a `sensor_msgs/msg/Image` stream — the standard ROS2 message type for camera frames.

This package is **not included in `ros-humble-desktop`**. It must be installed separately, making it a useful example of how to extend your ROS2 installation beyond what the desktop metapackage provides.

For this notebook we will use **Lenna**, a 512×512 pixel image that has been the de facto standard test image in image processing and computer vision since 1973. Its mix of smooth gradients, sharp edges, and fine texture makes it useful for benchmarking algorithms. You will find it referenced in virtually every image processing textbook.

In a real robotics system, a hardware camera driver would publish to the same `/image_raw` topic using the same `sensor_msgs/msg/Image` type. The downstream nodes — object detectors, visual odometry pipelines, tracking algorithms — subscribe to that topic and do not care whether the source is a real camera or `image_publisher`. The topic interface is the contract; the source is interchangeable.

## Step 1 — Install the Package

**Terminal 1** — install `image_publisher`:

```
sudo apt install ros-humble-image-publisher
```

If the package is already installed, apt will report `ros-humble-image-publisher is already the newest version` and exit cleanly — no action needed.

Verify that ROS2 can now find the package:

```
ros2 pkg list | grep image_publisher
```

Expected output:

```
image_publisher
```

If nothing appears, close the terminal and open a fresh one (with `source /opt/ros/humble/setup.bash`) — apt-installed packages are visible immediately in a new shell.

---

## Step 2 — Run the Node

**Terminal 1** — launch `image_publisher_node` with the Lenna test image and leave it running:

```
ros2 run image_publisher image_publisher_node \
  --ros-args -p filename:="$HOME/ros2_tutorial/module_01_basic_ros2_clis/Lenna_(test_image).png"
```

You should see log output like:

```
[INFO] [ImagePublisher]: File name for publishing image is : /home/<user>/ros2_tutorial/module_01_basic_ros2_clis/Lenna_(test_image).png
[INFO] [ImagePublisher]: Flip horizontal image is : false
```

The node is now publishing Lenna at 10 Hz. Leave Terminal 1 running and switch to Terminal 2.

---

## Step 3 — View the Image Stream

`showimage` is a viewer node from the `image_tools` package (already in `ros-humble-desktop`). By default it subscribes to the topic `/image`, so we remap that to `/image_raw` to match what `image_publisher_node` publishes.

**Terminal 2** — launch the viewer and leave it running:

```
ros2 run image_tools showimage --ros-args -r /image:=/image_raw
```

A window opens displaying the Lenna image, updating at 10 Hz.

This is also an example of topic remapping in action — exactly what you practiced in 1.9. `showimage`'s default subscription topic is `/image`, but `image_publisher_node` publishes to `/image_raw`. The `-r /image:=/image_raw` flag rewires the connection at launch time without touching either node's source code.
