## Answer 1 — Node Inventory


Expected output:

```
/robot/teleop_turtle
```

Two nodes, both in the `/robot` namespace.

**Package identification**: both belong to the `turtlesim` package. You can confirm this with `ros2 node info /robot/turtlesim` — the package is not printed directly, but you already launched them with `ros2 run turtlesim <executable>`. The package is always the first argument to `ros2 run`.

## Answer 2 — Topic Map

```bash
ros2 topic list -t
ros2 topic info /robot/turtle1/cmd_vel
ros2 topic info /robot/turtle1/pose
# ... repeat for other topics
```

| Topic | Type | Publisher | Subscriber |
|---|---|---|---|
| `/robot/turtle1/cmd_vel` | `geometry_msgs/msg/Twist` | `/robot/teleop_turtle` | `/robot/turtlesim` |
| `/robot/turtle1/pose` | `turtlesim/msg/Pose` | `/robot/turtlesim` | — |
| `/robot/turtle1/color_sensor` | `turtlesim/msg/Color` | `/robot/turtlesim` | — |

Topics with no subscribers: `/robot/turtle1/pose` and `/robot/turtle1/color_sensor` (dead sinks — hidden in `rqt_graph` by default).

Note: infrastructure topics (`/rosout`, `/parameter_events`) are omitted above for clarity.

## Answer 3 — Computation Graph

```
+----------------------+    /robot/turtle1/cmd_vel       +-------------------+
|  /robot/teleop_turtle| --(geometry_msgs/msg/Twist)--> |  /robot/turtlesim |
+----------------------+                                 +-------------------+
                                                                 |
                                          +----------------------+---------------------+
                                          |                                            |
                               /robot/turtle1/pose                /robot/turtle1/color_sensor
                               turtlesim/msg/Pose                 turtlesim/msg/Color
                                          |                                            |
                                   (no subscriber)                           (no subscriber)
```

**What `/robot/teleop_turtle` is doing**: it reads keyboard input and publishes velocity commands to `/robot/turtle1/cmd_vel`. It does **not** subscribe to anything from `/robot/turtlesim` — it has no feedback. This makes it an **open-loop publisher**: it sends commands without knowing where the turtle currently is or whether it reached any goal.

Compare this to a controller node, which would also subscribe to `/robot/turtle1/pose` to close the loop — adjusting its commands based on the turtle's actual position and heading.

## Answer 4 — Type Deep-Dive

```bash
ros2 interface show turtlesim/msg/Pose
```

Output:

```
float32 x
float32 y
float32 theta
float32 linear_velocity
float32 angular_velocity
```

- **`theta`**: the turtle's current heading in radians. `float32` confirms it is a continuous angle, not a discrete direction.
- **`linear_velocity` and `angular_velocity`**: the current speed of the turtle. `x` and `y` alone tell you where the turtle is — not whether it is moving or how fast. With velocity fields, a subscriber can detect motion or confirm the turtle has stopped without comparing consecutive position readings.

During a **forward key press**: `x` and `y` change, `theta` stays constant, `linear_velocity` is non-zero, `angular_velocity` is near zero.

During a **left/right key press**: `x` and `y` change slightly (the turtle arcs), `theta` changes, `angular_velocity` is non-zero, `linear_velocity` may be non-zero if the key also commands forward motion.

## Answer 5 — Spawn a Second Turtle

```bash
ros2 service list -t
ros2 interface show turtlesim/srv/Spawn
ros2 service call /robot/spawn turtlesim/srv/Spawn "{x: 3.0, y: 3.0, theta: 0.0, name: 'turtle2'}"
ros2 topic list -t
```

All services are provided by `/robot/turtlesim`. `/robot/teleop_turtle` exposes only the standard parameter services.

`/robot/spawn` type:

```
float32 x
float32 y
float32 theta
string name
---
string name
```

The caller provides position and optionally a name; the server returns the name it assigned. If `name` is left empty, turtlesim generates one automatically (`turtle2`, `turtle3`, …).

**New topics after spawning `turtle2`:**

```
/robot/turtle2/cmd_vel  [geometry_msgs/msg/Twist]   publisher: /robot/turtlesim   subscriber: —
/robot/turtle2/pose     [turtlesim/msg/Pose]         publisher: /robot/turtlesim   subscriber: —
/robot/turtle2/color_sensor [turtlesim/msg/Color]    publisher: /robot/turtlesim   subscriber: —
```

The single `turtlesim` node now manages both turtles. `/robot/turtle1/cmd_vel` is still the only topic with a subscriber — nothing yet sends commands to `turtle2`.

## Answer 6 — Parameter Inspection

```bash
ros2 param list /robot/turtlesim
```

```
/robot/turtlesim:
  background_b
  background_g
  background_r
  use_sim_time
```

Default background values: `background_r: 69`, `background_g: 86`, `background_b: 255`.

`background_r` is type `integer`, range 0–255, step 1 (confirmed with `ros2 param describe`).

Setting `background_r` to `200` changes the simulator background colour immediately — no restart needed. This is the defining property of a ROS2 parameter: runtime-configurable without stopping the node.

Only `use_sim_time` — teleop_turtle exposes no application-specific parameters.

## Answer 7 — Redirect Control to Turtle2

**How:** use topic remapping at launch time (covered in 1.9). The rule `turtle1/cmd_vel:=turtle2/cmd_vel` tells teleop to publish to `turtle2/cmd_vel` instead of its default `turtle1/cmd_vel`. The namespace is still `/robot`, so both resolve to `/robot/turtle2/cmd_vel`.

```bash
# Ctrl+C in Terminal 2, then:
ros2 run turtlesim turtle_teleop_key \
  --ros-args -r __ns:=/robot -r turtle1/cmd_vel:=turtle2/cmd_vel
```

**After rewiring:**

| Topic | Publisher count | Subscriber count |
|---|---|---|
| `/robot/turtle1/cmd_vel` | 0 | 1 (turtlesim still listens — nothing sends) |
| `/robot/turtle2/cmd_vel` | 1 (teleop) | 1 (turtlesim) |

**Final graph** (after Exercise 7, Dead Sinks visible):

```
+----------------------+    /robot/turtle2/cmd_vel       +-------------------+
|  /robot/teleop_turtle| --(geometry_msgs/msg/Twist)--> |  /robot/turtlesim |
+----------------------+                                 +-------------------+
                                                                 |
                          +----------+----------+----------+----+----+
                          |          |          |          |         |
              turtle1/cmd_vel   turtle1/pose  turtle1/color  turtle2/pose  turtle2/color_sensor
              (sub only,       (dead sink)   (dead sink)    (dead sink)   (dead sink)
               no publisher)
```

`/robot/turtle1/cmd_vel` appears in rqt_graph as a topic rectangle with only an incoming arrow from turtlesim's subscriber side and no publisher — the inverse of a dead sink. Pressing arrow keys now moves `turtle2`; `turtle1` stays still because no node publishes to its cmd_vel anymore.
