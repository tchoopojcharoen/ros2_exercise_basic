# 1.5 — Services (CLI)

## Learning Objectives

By the end of this section, you will be able to:
- List all active services with `ros2 service list`
- Inspect a service's type and interface definition with `ros2 service type`, `ros2 service info`, and `ros2 interface show`

## Services vs Topics — Recap

From 1.3: a service exists specifically because a response is required. The caller sends a **request**, blocks until the server processes it, then receives a **response**.

This is different from a topic, where the publisher broadcasts continuously and receives no acknowledgement. Services are one-to-one and synchronous: one call in, one reply out.

The CLI commands in this section let you inspect and call services without writing any code.

## `ros2 service list`

Lists every service currently available in the ROS2 system.

```
ros2 service list
```

You should see something like:

```
/clear
/kill
/reset
/spawn
/turtle1/set_pen
/turtle1/teleport_absolute
/turtle1/teleport_relative
/turtlesim/describe_parameters
/turtlesim/get_parameter_types
/turtlesim/get_parameters
/turtlesim/list_parameters
/turtlesim/set_parameters
/turtlesim/set_parameters_atomically
```

The `/turtlesim/..._parameters` services appear on every node — they are the standard parameter interface that ROS2 creates automatically. The services above them (`/clear`, `/kill`, `/reset`, `/spawn`, `/turtle1/...`) are specific to turtlesim.

Add `-t` to see the service type alongside each name:

```
ros2 service list -t
```

## `ros2 service type` and `ros2 service info`

`ros2 service type` prints the type of a single service. `ros2 service info` additionally shows which nodes are serving it.

```
ros2 service type /turtle1/teleport_absolute
```

```
ros2 service info /turtle1/teleport_absolute
```

Expected output from `service info`:

```
Type: turtlesim/srv/TeleportAbsolute
Clients:

Servers:
  /turtlesim
```

No clients are listed because nothing is currently calling this service. `/turtlesim` is the server — the node that will handle the call when one arrives.

---

## `ros2 interface show`

Shows the full field definition of a message or service type. For services, the output is split into two sections by `---`: the **request** fields above the line, and the **response** fields below it.

```
ros2 interface show turtlesim/srv/TeleportAbsolute
```

Output:

```
float32 x
float32 y
float32 theta
---
```

Three request fields, empty response. `TeleportAbsolute` moves the turtle to a position and returns nothing.

Compare this to a service that has a non-empty response:

```
ros2 interface show turtlesim/srv/Spawn
```

Output:

```
float32 x
float32 y
float32 theta
string name
---
string name
```

Four request fields, one response field. `/spawn` creates a new turtle and returns its name. The `---` line is always the boundary: everything above is what you send, everything below is what you get back.

This same `ros2 interface show` command works for message types too — `ros2 interface show turtlesim/msg/Pose` lists the fields you saw in `ros2 topic echo` output.

## `ros2 service call`

Calls a service from the command line. The syntax is:

```bash
ros2 service call <service_name> <service_type> "<request_yaml>"
```

Start with `/clear`, which takes an empty request and clears the turtle's trail:

```
ros2 service call /clear std_srvs/srv/Empty "{}"
```

The trail drawn by the turtle disappears. The response is empty, so the only feedback is the visual change in the simulator window.

Now call `/turtle1/teleport_absolute` to move the turtle to a specific position:

```
ros2 service call /turtle1/teleport_absolute turtlesim/srv/TeleportAbsolute "{x: 5.0, y: 5.0, theta: 0.0}"
```

The turtle jumps to the center of the window. Unlike `ros2 topic pub`, teleportation is instantaneous and leaves no trail.

As with `ros2 topic pub`, you only need to specify the fields you want to set — unspecified fields default to zero. `"{x: 5.0, y: 5.0}"` is valid and leaves `theta` at `0.0`.

Now call `/spawn` to add a second turtle and observe the response:

```
ros2 service call /spawn turtlesim/srv/Spawn "{x: 2.0, y: 2.0, theta: 0.0, name: 'turtle2'}"
```

A second turtle appears in the simulator window. The terminal prints the response:

```
requester: making request: turtlesim.srv.Spawn_Request(x=2.0, y=2.0, theta=0.0, name='turtle2')

response:
turtlesim.srv.Spawn_Response(name='turtle2')
```

This is the only CLI command so far that shows a non-empty response — the server returns the name it assigned to the new turtle. In code, a service client receives this response object and can act on it.
