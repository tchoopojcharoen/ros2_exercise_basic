## What is rqt_graph?

`rqt_graph` is a GUI tool that draws the live computation graph of a running ROS2 system — the same structure you explored through the CLI in 1.3 and 1.4, now rendered as a visual diagram.

It is part of `rqt`, a plugin-based Qt framework for ROS2 GUI tools. `rqt_graph` is one of many rqt plugins; others include `rqt_console` (log viewer), `rqt_plot` (time-series plotting), and `rqt_topic` (topic inspection).

> If `rqt_graph` is not installed, run:
> ```bash
> sudo apt install ros-humble-rqt-graph
> ```

## Step 1 — Refresh the Graph

When `rqt_graph` opens, it may show an empty or stale graph. **Click the refresh button** (the circular arrow icon in the top-left toolbar) to query the current system state.

After refreshing, you should see two nodes connected by a topic:

```
/teleop_turtle  ──/turtle1/cmd_vel──>  /turtlesim
```

Nodes are drawn as ellipses. Topics are drawn as rectangles (in the default Nodes/Topics view). Arrows show the direction of data flow — from publisher to subscriber.

> **Note:** `rqt_graph` does not auto-refresh. Any time you start or stop a node, click refresh again to update the view.

---

## Step 2 — Display Mode

The dropdown in the top-left controls what the graph shows:

| Mode | What is drawn |
|---|---|
| **Nodes only** | Nodes as ellipses; topics shown as edge labels, not as separate shapes |
| **Nodes/Topics (All)** | Nodes and topics both drawn as distinct shapes |
| **Topics only** | Only topics; nodes are omitted |

Switch to **Nodes/Topics (All)** and refresh. The graph now shows topic rectangles as explicit nodes in the diagram, making the data flow easier to read when many topics are involved.

---

## Step 3 — Reveal Hidden Topics

By default, `rqt_graph` hides two categories of topics that would otherwise clutter the diagram:

**Dead sinks** — topics that are published to but have no subscribers. In the current setup, `/turtle1/pose` and `/turtle1/color_sensor` are published by turtlesim but nothing subscribes to them yet, so they are hidden.

**Leaf topics** — topics at the edge of the graph that are not consumed by any other node to produce further output. Infrastructure topics such as `/rosout` and `/parameter_events` fall into this category.

To reveal them:
1. Look for the **Hide** checkboxes in the toolbar (labelled "Dead Sinks" and "Leaf Topics").
2. Uncheck both boxes.
3. Click refresh.

The graph now shows the full picture:

```
/teleop_turtle  ──/turtle1/cmd_vel──>  /turtlesim  ──/turtle1/pose──>  (no subscriber)
                                                   ──/turtle1/color_sensor──>  (no subscriber)
                                                   ──/rosout──>  ...
```

Compare this to the reference diagram you drew in 1.3. The topics that had no subscriber in your CLI exploration appear here as dead-end arrows.

---

## Step 4 — Observe a Live Graph Change

The graph updates to reflect the live system. To observe this:

1. Go to Terminal 2 (where `turtle_teleop_key` is running) and press `Ctrl+C` to stop it.
2. Switch back to `rqt_graph` and click refresh.

The `/teleop_turtle` node and the `/turtle1/cmd_vel` connection disappear — only `/turtlesim` remains.

3. Restart teleop in Terminal 2:

```bash
ros2 run turtlesim turtle_teleop_key
```

4. Click refresh in `rqt_graph`. The connection reappears.

This reflects a key property of the ROS2 computation graph: it is dynamic. Nodes can join and leave at any time, and the graph changes accordingly.

## Step 5 — Namespaces in the Graph

From 1.9, you know that launching a node with `__ns:=` prefixes all its names. `rqt_graph` makes this structural separation visible at a glance.

Stop the current processes first:
- `Ctrl+C` in Terminal 1 (turtlesim)
- `Ctrl+C` in Terminal 2 (teleop)
- Leave Terminal 3 (rqt_graph) open

Now launch two turtlesim instances in separate namespaces:

**Terminal 1:**

```bash
ros2 run turtlesim turtlesim_node --ros-args -r __ns:=/sim1
```

**Terminal 2:**

```bash
ros2 run turtlesim turtlesim_node --ros-args -r __ns:=/sim2
```

Two simulator windows open. Click refresh in `rqt_graph`.

You should see two completely separate subgraphs — one rooted at `/sim1/turtlesim`, the other at `/sim2/turtlesim`. Their topics share no connections:

```
/sim1/turtlesim   (publishes /sim1/turtle1/pose, /sim1/turtle1/color_sensor, ...)
/sim2/turtlesim   (publishes /sim2/turtle1/pose, /sim2/turtle1/color_sensor, ...)
```

This is the graphical confirmation of what you observed with `ros2 topic list` in 1.9: namespaces produce completely independent sets of names.

### Namespace filter

When many namespaced nodes are running, the graph can become dense. Use the **namespace filter** text field in the top-left toolbar to restrict the view to one namespace. Typing `/sim1` shows only nodes and topics whose names begin with `/sim1`, hiding everything from `/sim2`.

This filter is purely visual — it does not affect the running system.
