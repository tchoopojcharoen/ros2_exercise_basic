## Actions vs Services — Recap

From 1.3: a service completes immediately — one request in, one response out. An action handles tasks that take time. The caller sends a **goal**, the server executes it over multiple steps, and the caller receives **feedback** while waiting and a final **result** when done. The caller can also **cancel** the goal mid-execution.

| | Service | Action |
|---|---|---|
| Duration | Completes immediately | Takes time |
| Intermediate updates | None | Feedback |
| Cancellable | No | Yes |
| Interface sections | Request / Response | Goal / Result / Feedback |

Turtlesim has one action: `/turtle1/rotate_absolute`, which rotates the turtle to a target heading. This takes time to execute and streams the remaining angle as feedback — making it a good example to observe all three phases.

Lists every action server currently available in the ROS2 system.

```
ros2 action list
```

You should see:

```

```

Turtlesim provides only one action. Add `-t` to see its type:

```
ros2 action list -t
```

## `ros2 action info`

Shows which nodes are acting as the server and any current clients.

```
ros2 action info /turtle1/rotate_absolute
```

Expected output:

```
Action: /turtle1/rotate_absolute
Action clients: 0
Action servers: 1
    /turtlesim
```

The `/turtlesim` node is the server. No clients are connected yet.

## `ros2 interface show` for Actions

Action interface definitions have three sections separated by `---`: **Goal**, **Result**, and **Feedback**.

```
ros2 interface show turtlesim/action/RotateAbsolute
```

Output:

```
# Goal
float32 theta
---
# Result
float32 delta
---
# Feedback
float32 remaining
```

- **Goal** (`theta`): the target heading in radians
- **Result** (`delta`): the total angle rotated once the goal is reached
- **Feedback** (`remaining`): the angle still to rotate, streamed during execution

Compare this to the service interface format from 1.5, which only had two sections (request / response). Actions add a third section for feedback.

## `ros2 action send_goal`

Sends a goal to an action server. The syntax is:

```bash
ros2 action send_goal <action_name> <action_type> "<goal_yaml>"
```

Send a goal to rotate the turtle to `theta = 1.57` radians (~90°):

```
ros2 action send_goal /turtle1/rotate_absolute turtlesim/action/RotateAbsolute "{theta: 1.57}"
```

The turtle rotates to face ~90°. The output looks like:

```
Waiting for an action server to become available...
Sending goal:
     theta: 1.57

Goal accepted with ID: ...

Result:
    delta: -1.5599...

Goal finished with status: SUCCEEDED
```

The command blocks until the goal finishes, then prints the result. Note that without `--feedback`, the intermediate feedback messages are not shown — you only see the final result.

### With feedback

Add `--feedback` to print each feedback message as it arrives. Rotate to a different angle so the motion is visible:

```
ros2 action send_goal --feedback /turtle1/rotate_absolute turtlesim/action/RotateAbsolute "{theta: 0.0}"
```

The output now includes feedback messages while the turtle rotates:

```
Sending goal:
     theta: 0.0

Goal accepted with ID: ...

Feedback:
    remaining: 1.4

Feedback:
    remaining: 0.8

Feedback:
    remaining: 0.2

Result:
    delta: -1.57...

Goal finished with status: SUCCEEDED
```

`remaining` counts down from the initial angle difference to zero as the turtle turns.

### Cancellation

Because actions are designed for long-running tasks, goals can be cancelled mid-execution. To observe this, send a goal with a large rotation and press `Ctrl+C` before it finishes:

```bash
ros2 action send_goal --feedback /turtle1/rotate_absolute turtlesim/action/RotateAbsolute "{theta: 10.0}"
```

Pressing `Ctrl+C` sends a cancel request to the server. The turtle stops mid-rotation and the goal status changes to `CANCELED`. This is the key behavioral difference from a service: a service call cannot be cancelled once the server has accepted it.
