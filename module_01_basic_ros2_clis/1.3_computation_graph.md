## What is a Computation Graph?

When a ROS2 system is running, every process that participates in communication becomes a **node**. Nodes exchange data through **topics**, **services**, and **actions**. Together, these nodes and connections form a directed graph — the **computation graph**.

- **Nodes** are the processes doing computation
- **Topics** are named channels for continuous, one-to-many data streams
- **Services** are named endpoints for on-demand request/response calls
- **Actions** are named endpoints for long-running requests with progress feedback and cancellation

Understanding this graph — and being able to inspect it — is one of the most powerful skills in ROS2 development.

## Packages

A **package** is the basic unit of software distribution in ROS2. It is a directory that bundles together any combination of:

- **Nodes** — the executable programs
- **Message, service, and action type definitions** — the schemas for data exchanged over topics, services, and actions
- **Launch files and configuration** — covered in later modules

Everything you run or reference by name in ROS2 belongs to a package.

### Packages in `ros2 run`

The command structure makes the package explicit:

```bash
ros2 run <package> <executable>
```

When you ran `ros2 run turtlesim turtlesim_node` in 1.2:
- `turtlesim` is the **package**
- `turtlesim_node` is an **executable** inside that package

One package can contain multiple executables. The `turtlesim` package provides both `turtlesim_node` and `turtle_teleop_key`.

### Packages in type names

Every message, service, and action type name starts with the package that defines it, in the form `<package>/<category>/<TypeName>`:

| Type string | Package | Category | Type |
|---|---|---|---|
| `geometry_msgs/msg/Twist` | `geometry_msgs` | `msg` | `Twist` |
| `turtlesim/msg/Pose` | `turtlesim` | `msg` | `Pose` |
| `turtlesim/srv/Spawn` | `turtlesim` | `srv` | `Spawn` |

You will see these type strings in the output of `ros2 node info`, `ros2 topic list -t`, and `ros2 interface show` throughout this module. The package prefix tells you immediately which package to look in if you need the full definition.

> **Note:** A package does not have to contain any nodes. `geometry_msgs`, `std_msgs`, and `sensor_msgs` ship only type definitions — no executables. Packages that contain only nodes and no type definitions are equally valid.

Creating and building your own packages is covered in Module 2. For now, treat a package as the named container that everything in ROS2 belongs to.

## Core Building Blocks

| Concept | What It Is | Direction |
|---|---|---|
| **Node** | A running process (your Python/C++ program) | — |
| **Topic** | A named channel for a continuous data stream | One-to-many (broadcast) |
| **Service** | A named call that returns a result | One-to-one (request/response) |
| **Parameter** | A named configuration value on a specific node | Read/write at runtime |
| **Action** | A long-running task that reports progress | One-to-one + feedback |

### Key distinctions

**Subscriber vs Service:**
- A subscriber reacts to incoming data — fire-and-forget, no acknowledgement is sent back to the publisher
- A service exists specifically because a response is required — the caller blocks and waits for that response before continuing

**Services vs Actions:**
- A service completes immediately — one request in, one response out
- An action handles tasks that take time — it accepts a goal, sends periodic feedback while working, and returns a final result; the caller can also cancel it mid-way

### When to use each

ROS2 communication patterns map onto two triggering paradigms:

| Trigger | Completes quickly | Long-running / cancellable |
|---|---|---|
| Time-triggered | Timer callback (inside a node) | — |
| Event-triggered, no response needed | Subscriber callback | — |
| Event-triggered, response needed | Service | Action |

- **Time-triggered** systems run on a fixed schedule — a control loop updating at 10 Hz is the classic example
- **Event-triggered** systems react to external stimuli — an incoming message, an external request, or a goal being sent

## CLI Tools: Inspecting Nodes

### `ros2 run`

Launches an executable from a ROS2 package (see the Packages section above):

```bash
ros2 run <package_name> <executable_name>
```

Example: `ros2 run turtlesim turtlesim_node`
- Package: `turtlesim`
- Executable: `turtlesim_node`

### `ros2 node list`

Lists all currently running nodes:

```bash
ros2 node list
```

### `ros2 node info <node_name>`

Shows everything a node publishes, subscribes to, serves, and its parameters:

```bash
ros2 node info /turtlesim
```

In Terminal 3, try these commands:

```
ros2 node list
```

```
ros2 node info /turtlesim
```

```
ros2 node info /teleop_turtle
```

### Reading `ros2 node info` output

The output is organized into sections:

```
/turtlesim
  Subscribers:
    /turtle1/cmd_vel: geometry_msgs/msg/Twist       <-- listens for velocity commands
  Publishers:
    /turtle1/pose: turtlesim/msg/Pose               <-- broadcasts turtle position
    /turtle1/color_sensor: turtlesim/msg/Color      <-- broadcasts background color under turtle
    /rosout: rcl_interfaces/msg/Log                 <-- ROS2 logging (every node has this)
  Service Servers:
    /turtle1/teleport_absolute: turtlesim/srv/TeleportAbsolute
    /turtle1/teleport_relative: turtlesim/srv/TeleportRelative
    /turtle1/set_pen: turtlesim/srv/SetPen
    /reset: std_srvs/srv/Empty
    /clear: std_srvs/srv/Empty
    /spawn: turtlesim/srv/Spawn
    /kill: turtlesim/srv/Kill
    ... (parameter services — every node has these)
```

Each topic line shows: `topic_name: message_type`
Each service line shows: `service_name: service_type`

### Things to notice

1. `/teleop_turtle` only **publishes** — it never subscribes to anything
2. `/turtlesim` only **subscribes** to `cmd_vel` — the command input
3. `/turtlesim` **publishes** pose and color sensor data continuously
4. No node currently subscribes to `/turtle1/pose` — but any node you write can
5. Services are provided by `/turtlesim` and called by external code (or CLI)

## Summary

- A **package** is the unit of software distribution in ROS2 — it bundles nodes, type definitions, and configuration together
- The **computation graph** is a live wiring diagram of running ROS2 nodes and their connections
- **Topics** are broadcast channels (pub/sub, one-to-many); **services** are request/response calls (one-to-one); **actions** are long-running tasks with feedback and cancellation
- `ros2 node list` shows running nodes; `ros2 node info` shows a node's full interface
- The turtlesim graph has 2 nodes, 3 topics, and 7+ services
