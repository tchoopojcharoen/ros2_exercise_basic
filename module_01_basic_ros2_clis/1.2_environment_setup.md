## Step 1 — Confirm Your Operating System

Although ROS2 supports Windows 11, and Mac, some grpahical tools are mostly stable on Linux. Ubuntu is the recommended distribution. 

In a terminal, run:

```
uname -a
```

You should see something like:
```
Linux hostname 6.8.0-xx-generic #xx-Ubuntu SMP ... x86_64 x86_64 x86_64 GNU/Linux
```

You can also use a Linux virtual machine or Docker container that has ROS2 installed.

---

## Step 2 — Check Your ROS2 Installation

ROS2 installs into `/opt/ros/`. Each installed distribution gets its own subdirectory.

In a terminal, run:

```
ls /opt/ros/
```

You should see one or more distro names, for example:
```
humble
```

If the directory does not exist or is empty, ROS2 is not installed. Follow the [official ROS2 installation guide](https://docs.ros.org/en/rolling/Installation.html) before continuing.

> **Note:** If you have multiple distros installed (e.g., `humble` and `jazzy`), take note of which one you want to use. The cells below use `DISTRO=humble` — change that value to match your chosen distro.

This only confirms the directory exists — not that ROS2 is actually functional. Let's verify the environment is active.

The cell below sources ROS2 and checks the `ROS_DISTRO` environment variable. Both commands are in the same cell because each cell runs in an isolated subshell — sourcing in one cell does not carry over to the next.

```
DISTRO=humble   # change this if your distro is different
source /opt/ros/$DISTRO/setup.bash && printenv ROS_DISTRO
```

You should see your distro name printed, for example:
```
humble
```

If nothing is printed, or you see `ros2: command not found`, ROS2 is not installed or the path is wrong.

Now do a final check by launching turtlesim. This confirms that ROS2 is fully functional and that the desktop packages are installed:

```
DISTRO=humble   # change this if your distro is different
source /opt/ros/$DISTRO/setup.bash
ros2 run turtlesim turtlesim_node
```

A window with a blue background and a turtle in the center should appear — this is the turtlesim simulator.

> **Note:** This command keeps running until turtlesim is closed. Press `Ctrl+C` in the terminal to stop it.

If you see an error like `package 'turtlesim' not found`, you have the base ROS2 install rather than the full desktop. Fix it with:
```bash
sudo apt install ros-humble-desktop
```
replacing `humble` with your distro name if different.

## Step 3 — Understanding `.bashrc`

Before fixing the ROS2 setup, you need to understand `.bashrc`.

### What is `.bashrc`?

`.bashrc` is a **shell startup script**. It is a plain text file located at `~/.bashrc` (the `~` means your home directory, e.g. `/home/yourname/.bashrc`).

**Bash runs `.bashrc` automatically every time you open a new terminal.** This makes it the right place to put things you always want available:
- Custom aliases (e.g., `alias ll='ls -la'`)
- Environment variables (e.g., `export EDITOR=vim`)
- Path additions (e.g., `export PATH=$PATH:/my/tools`)
- Shell setup commands (e.g., activating a Python virtual environment)

### Why does it matter for ROS2?

ROS2 is not globally available by default. It installs its tools and libraries into `/opt/ros/<distro>/`, but the shell does not know about them until you explicitly **source** the setup file:

```bash
source /opt/ros/<distro>/setup.bash
```

Sourcing a file means *running it in the current shell's context*, so that any environment variables it sets (like `PATH`, `AMENT_PREFIX_PATH`, `ROS_DISTRO`) take effect in your current terminal.

Without sourcing, running `ros2` in a terminal produces:
```
ros2: command not found
```

With sourcing, `ros2` is available, packages are discoverable, and everything works.

---

## Step 4 — The Sourcing Problem

The obvious solution is to source ROS2 manually in every terminal:

```bash
source /opt/ros/<distro>/setup.bash
```

This works — but you have to remember to do it in **every single terminal** you open. A typical ROS2 session involves 3–5 terminals. Forgetting to source in one of them causes confusing errors.

The fix: add the source command to `.bashrc` so it runs automatically.

---

## Step 5 — The Permanent Fix

The command below appends the ROS2 source line to your `.bashrc`, but only if it is not already there (so running it multiple times is safe):

```
DISTRO=humble   # change this if your distro is different
grep -q 'source /opt/ros' ~/.bashrc \
    && echo 'Already in .bashrc — no change made.' \
    || (echo "source /opt/ros/$DISTRO/setup.bash" >> ~/.bashrc \
        && echo 'Added to .bashrc successfully.')
```

Let's verify what was added by looking at the last few lines of `.bashrc`:

```
tail -5 ~/.bashrc
```

You should see a line like:
```
source /opt/ros/humble/setup.bash
```

> **Important:** `>>` appends to a file. `>` would overwrite it. Always use `>>` when adding to `.bashrc`.

### Why can't I just run `source ~/.bashrc` here?

Each shell cell in Jupyter spawns a **new subshell**, runs the command, then discards the shell. Any environment variable changes made inside that subshell are lost when it exits — they don't carry back into the Jupyter kernel's environment.

To apply the `.bashrc` change, you need to **open a new terminal**. New terminals start a fresh bash session, which automatically runs `.bashrc`.

## Step 6 — Smoke Test

Open a **new terminal** (not this notebook) and run:

```bash
ros2 run turtlesim turtlesim_node
```

If a window appears with a turtle in it, your ROS2 installation is working correctly.

You can also check that the `ROS_DISTRO` environment variable is now set by running:

```bash
printenv ROS_DISTRO
```

This should print your distro name (e.g., `humble`).

Close turtlesim for now (`Ctrl+C` in the terminal). You will launch it again in the next section.
