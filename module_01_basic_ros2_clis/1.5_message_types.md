## The Problem

In 1.4 you published to `/turtle1/cmd_vel` using:

```bash
ros2 topic pub --once /turtle1/cmd_vel geometry_msgs/msg/Twist "{linear: {x: 1.0}}"
```

Two things in that command required prior knowledge: the **type name** (`geometry_msgs/msg/Twist`) and the **YAML structure** (`{linear: {x: 1.0}}`). How do you find those for an unfamiliar topic?

There are two ways:

1. **From a running network** — inspect a topic that is already active
2. **From documentation** — look it up online, then verify with the CLI

This notebook covers both.

## Path 1 — Discovering the Type from a Running Network

When a topic is active, the type string is visible in two places.

### From `ros2 topic list -t`

The `-t` flag appends the type in brackets after each topic name:

```
ros2 topic list -t
```

Expected output:

```
/turtle1/cmd_vel [geometry_msgs/msg/Twist]
/turtle1/color_sensor [turtlesim/msg/Color]
/turtle1/pose [turtlesim/msg/Pose]
...
```

The bracketed string is the full type name. Its format is always:

```
<package>/<category>/<TypeName>
```

### From `ros2 topic info`

For a single topic, `ros2 topic info` shows the type along with publisher and subscriber counts:

```
ros2 topic info /turtle1/cmd_vel
```

Expected output:

```
Type: geometry_msgs/msg/Twist
Publisher count: 0
Subscription count: 1
```

Both commands give you the same type string. Once you have it, you can read the definition — covered in the next section.

## Path 2 — Finding Types from Documentation

Not every type you need will be visible on the network. When you are writing code that must publish or subscribe to a well-known type — for example, `sensor_msgs/msg/Image` — you look it up first rather than waiting for a live topic.

**Official sources:**

- **[docs.ros.org](https://docs.ros.org)** — the main ROS2 documentation site. Each common package (`geometry_msgs`, `sensor_msgs`, `nav_msgs`) has an API page listing every type with field descriptions.
- **Package README files** — robot hardware packages and third-party ROS2 packages typically document the topics they publish and subscribe to, including type names.

Once you have the type name from documentation, you use `ros2 interface show` (covered below) to read the exact field definitions locally.

### Browsing what is installed

You can also browse locally. To list every package that defines interfaces:

```
ros2 interface packages
```

To list all interfaces in a specific package:

```
ros2 interface package turtlesim
```

Expected output:

```
turtlesim/action/RotateAbsolute
turtlesim/msg/Color
turtlesim/msg/Pose
turtlesim/srv/Kill
turtlesim/srv/SetPen
turtlesim/srv/Spawn
turtlesim/srv/TeleportAbsolute
turtlesim/srv/TeleportRelative
```

Notice the three categories in the results:

| Category | Used for |
|---|---|
| `msg` | Topic data |
| `srv` | Service request and response |
| `action` | Action goal, result, and feedback |

## Reading the Type Definition with `ros2 interface show`

Regardless of which path you took to get the type string, the next step is always the same: pass the type string to `ros2 interface show` to read its field definitions.

### A simple message

Start with the type you found in the exercise above:

```
ros2 interface show turtlesim/msg/Pose
```

Output:

```
float32 x
float32 y
float32 theta
float32 linear_velocity
float32 angular_velocity
```

Each line is `<type> <field_name>`. All five fields are `float32` — a 32-bit floating-point number. This is a flat message with no nesting.

### Primitive types

ROS2 has a fixed set of built-in types:

| Type | Size |
|---|---|
| `bool` | 1 bit |
| `int8`, `int16`, `int32`, `int64` | Signed integers |
| `uint8`, `uint16`, `uint32`, `uint64` | Unsigned integers |
| `float32`, `float64` | Floating-point |
| `string` | UTF-8 text |

Any field whose type is **not** in this table is itself another message — a nested type.

### A nested message

Now read the type used by `/turtle1/cmd_vel`:

```
ros2 interface show geometry_msgs/msg/Twist
```

Output:

```
# This expresses velocity in free space broken into its linear and angular parts.
Vector3  linear
        float64 x
        float64 y
        float64 z
Vector3  angular
        float64 x
        float64 y
        float64 z
```

`Twist` has two fields (`linear` and `angular`), both of type `Vector3`. Because `Vector3` is not a primitive, `ros2 interface show` expands it inline and indented so you can see the full structure without a separate lookup.

This directly explains the YAML you used in 1.4:

```yaml
{linear: {x: 1.0}}
```

`linear` maps to the first `Vector3` field; `x` maps to its `x` subfield. The remaining seven fields (`linear.y`, `linear.z`, `angular.x`, `angular.y`, `angular.z`) were omitted and defaulted to zero.

### Arrays

Fields can be arrays. A variable-length array is written as `type[]`, and a fixed-length array as `type[N]`. For example, the raw pixel data in `sensor_msgs/msg/Image` is:

```
uint8[] data
```

You will encounter array fields when working with sensors in later modules.

## Summary

**Finding the type string:**
- From a running topic: `ros2 topic list -t` or `ros2 topic info <topic>`
- From documentation: [docs.ros.org](https://docs.ros.org) or the package README
- Browsing installed packages: `ros2 interface package <package>`

**Reading the definition:**
- `ros2 interface show <type>` — always the next step once you have the type string

Service and action type definitions (which use `---` separators) are covered in 1.6 and 1.7 respectively.
