## What Is a Namespace?

Every node, topic, service, and parameter in ROS2 has a name. By default those names are global — `/turtlesim`, `/turtle1/cmd_vel`, `/turtle1/pose`. This works fine for a single robot, but breaks down as soon as you run the same software on two robots at once: both would try to own `/turtlesim`, publish to `/turtle1/cmd_vel`, and read from `/turtle1/pose`, causing conflicts.

A **namespace** is a prefix applied to a node and everything it owns. Launching `/turtlesim` into the namespace `/robot1` gives you:

```
node:    /robot1/turtlesim
topics:  /robot1/turtle1/cmd_vel
         /robot1/turtle1/pose
         /robot1/turtle1/color_sensor
```

A second instance in `/robot2` has its own fully separate set of names. The two instances cannot interfere with each other.

## Launching with a Namespace

Stop turtlesim in Terminal 1 (`Ctrl+C`), then relaunch it inside the namespace `/sim1`:

```bash
ros2 run turtlesim turtlesim_node --ros-args -r __ns:=/sim1
```

The `--ros-args` flag signals that what follows are ROS2 arguments rather than program arguments. `-r __ns:=/sim1` is the remap rule that sets the namespace.

Now run the same inspection commands:

```
ros2 node list
```

```
ros2 topic list
```

Expected output:

```
# ros2 node list
/sim1/turtlesim

# ros2 topic list
/sim1/turtle1/cmd_vel
/sim1/turtle1/color_sensor
/sim1/turtle1/pose
...
```

Every name is now prefixed with `/sim1/`. The node code itself did not change — the namespace was applied entirely from the outside at launch time.

## Two Instances, Two Namespaces

Stop the `/sim1` instance in Terminal 1 (`Ctrl+C`). Now launch two independent turtlesim instances:

**Terminal 1:**

```bash
ros2 run turtlesim turtlesim_node --ros-args -r __ns:=/sim1
```

**Terminal 2** — open a third terminal for this, and use Terminal 2 only for inspection commands:

```bash
ros2 run turtlesim turtlesim_node --ros-args -r __ns:=/sim2
```

Two simulator windows open. In a new terminal (Terminal 3 for commands, or reuse Terminal 2), list all nodes and topics:

```
ros2 node list
```

```
ros2 topic list
```

Expected output:

```
# ros2 node list
/sim1/turtlesim
/sim2/turtlesim

# ros2 topic list
/sim1/turtle1/cmd_vel
/sim1/turtle1/pose
/sim2/turtle1/cmd_vel
/sim2/turtle1/pose
...
```

Both instances coexist without conflict. Publishing to `/sim1/turtle1/cmd_vel` moves only the first turtle; `/sim2/turtle1/cmd_vel` controls only the second.

Stop both instances (`Ctrl+C` in both terminals) before continuing.

## What Is Remapping?

Namespaces apply a prefix to **everything** a node owns. Remapping is more surgical — it redirects a **specific** name to a different name, without affecting anything else.

The general syntax is:

```bash
ros2 run <package> <executable> --ros-args -r <from>:=<to>
```

Two names have special meaning:

| Special name | What it remaps |
|---|---|
| `__node` | The node's own name |
| `__ns` | The node's namespace (as used above) |

All other names refer to specific topics, services, or parameters by their full path.

---

## Remapping the Node Name

Launch turtlesim in Terminal 1 with a custom node name:

```bash
ros2 run turtlesim turtlesim_node --ros-args -r __node:=my_turtlesim
```

```
ros2 node list
```

Expected output:

```
/my_turtlesim
```

The executable is still `turtlesim_node` from the `turtlesim` package, but the running node identifies itself as `/my_turtlesim`. The topics remain unchanged (`/turtle1/cmd_vel` etc.) because only the node name was remapped.

Stop this instance before the next exercise.

## Remapping a Topic

Topic remapping redirects a specific topic to a different name. This is useful when two nodes use different names for the same logical channel and you want to connect them without modifying their code.

Launch turtlesim in Terminal 1 with `/turtle1/cmd_vel` remapped to `/robot/cmd_vel`:

```bash
ros2 run turtlesim turtlesim_node --ros-args -r turtle1/cmd_vel:=robot/cmd_vel
```

```
ros2 topic list
```

The topic `/robot/cmd_vel` now appears in place of `/turtle1/cmd_vel`. Publishing to `/robot/cmd_vel` will move the turtle. All other topics (`/turtle1/pose`, `/turtle1/color_sensor`) are unchanged.

Multiple remaps can be chained in a single launch:

```bash
ros2 run turtlesim turtlesim_node --ros-args \
  -r __ns:=/sim1 \
  -r __node:=my_sim \
  -r turtle1/cmd_vel:=robot/cmd_vel
```

Each `-r` flag is an independent remap rule. Namespace and name remaps apply simultaneously.

Stop turtlesim before continuing.

---

## Summary

| Concept | Syntax | What it changes |
|---|---|---|
| Namespace | `--ros-args -r __ns:=/prefix` | Prefixes **all** names the node owns |
| Node name remap | `--ros-args -r __node:=new_name` | The node's own name only |
| Topic / service remap | `--ros-args -r old/name:=new/name` | One specific name |

Both mechanisms are applied at launch time in the terminal — no source code change is needed. In later modules, launch files automate these arguments so you do not have to type them manually every time.
