## Topics as Channels

In 1.3 you saw topics appear in `ros2 node info` output — named channels that nodes use to broadcast data. This section teaches you to inspect those channels directly, without writing any code.

A topic has three observable properties:

| Property | What it tells you |
|---|---|
| **Name** | The channel identifier (e.g. `/turtle1/pose`) |
| **Type** | The message schema (e.g. `turtlesim/msg/Pose`) |
| **Connections** | Which nodes publish to it and which subscribe |

All four commands below work on a live system — they require at least one ROS2 node to be running.

## `ros2 topic list`

Lists every topic currently active in the ROS2 system.

```
ros2 topic list
```

You should see:

```
/parameter_events
/rosout
/turtle1/cmd_vel
/turtle1/color_sensor
/turtle1/pose
```

`/parameter_events` and `/rosout` appear on every ROS2 system — they are infrastructure topics created automatically by the middleware. The three `/turtle1/...` topics are specific to turtlesim.

Add `-t` to also see the message type next to each topic name:

```
ros2 topic list -t
```

## `ros2 topic info`

Shows a single topic's message type and the number of nodes publishing and subscribing to it.

```
ros2 topic info /turtle1/pose
```

Expected output:

```
Type: turtlesim/msg/Pose
Publisher count: 1
Subscription count: 0
```

Subscription count is 0 because nothing in this session subscribes to pose yet — you will write a subscriber node in a later module.

The type `turtlesim/msg/Pose` names the message schema. To see the full field definition of any message type, use `ros2 interface show`:

```bash
ros2 interface show turtlesim/msg/Pose
```

This is covered in 1.5 — Message Types.

Add `--verbose` to expand each publisher and subscriber into a full entry with its **QoS profile**:

```
ros2 topic info --verbose /turtle1/pose
```

Expected output:

```
Type: turtlesim/msg/Pose
Publisher count: 1

Node name: turtlesim
Node namespace: /
Topic type: turtlesim/msg/Pose
Endpoint type: PUBLISHER
GID: 01.0f.9e.e7.f3.3c.68.a4.00.00.00.00.00.00.1c.03.00.00.00.00.00.00.00.00
QoS profile:
  Reliability: RELIABLE
  History (Depth): KEEP_LAST (10)
  Durability: VOLATILE
  Lifespan: Infinite
  Deadline: Infinite
  Liveliness: AUTOMATIC
  Liveliness lease duration: Infinite

Subscription count: 0
```

QoS (Quality of Service) settings control how the middleware delivers messages — whether delivery is guaranteed (`Reliability`), how many messages are buffered (`History`), and whether late-joining subscribers receive messages published before they connected (`Durability`). Mismatched QoS between a publisher and a subscriber is a common source of silent connection failures in ROS2.

QoS profiles are covered in depth in a later module. For now, `--verbose` is how you inspect them from the CLI.

## `ros2 topic echo`

Prints every message published on a topic in real time.

> **Note:** This command will block your terminal — press `Ctrl+C` to stop it before moving on.

```
ros2 topic echo /turtle1/pose
```

You should see a continuous stream:

```
x: 5.544444561004639
y: 5.544444561004639
theta: 0.0
linear_velocity: 0.0
angular_velocity: 0.0
---
x: 5.544444561004639
...
```

Each `---` separator marks the boundary between messages. Notice `linear_velocity` and `angular_velocity` are zero — the turtle is stationary. You will see these change when you run `ros2 topic pub` below.

## `ros2 topic hz`

Measures how frequently messages are published on a topic — the publish rate in Hz.

> **Note:** This command will block your terminal — press `Ctrl+C` to stop it before moving on.

```
ros2 topic hz /turtle1/pose
```

Expected output (approximate):

```
average rate: 62.500
	min: 0.016s max: 0.016s std dev: 0.00009s window: 64
```

Turtlesim publishes pose at ~62.5 Hz. Knowing the publish rate matters for control loops: your controller must run at least as fast as the sensor data it consumes, otherwise it acts on stale values. A 10 Hz controller consuming 62 Hz pose data is fine; a 100 Hz controller consuming 10 Hz data will loop on duplicates.

To also measure bandwidth (bytes per second):

> **Note:** This command will block your terminal — press `Ctrl+C` to stop it before moving on.

```
ros2 topic bw /turtle1/pose
```

Pose messages are small — bandwidth is rarely a concern for local topics. It becomes relevant when bridging over a network (robot to base station) or streaming dense data such as point clouds or camera images.

## `ros2 topic pub`

Publishes a message to a topic from the command line — no node required.

The `--once` flag publishes one message and exits. In your terminal, run the following and watch the turtle move:

```
ros2 topic pub --once /turtle1/cmd_vel geometry_msgs/msg/Twist "{linear: {x: 1.0}}"
```

The turtle should nudge straight forward. Unspecified fields default to zero, so `{linear: {x: 1.0}}` is equivalent to writing out all six fields with `y`, `z`, and all angular components set to `0.0`. You only need to specify the fields you want to change from their default.

To publish continuously at 1 Hz instead:

```bash
ros2 topic pub --rate 1 /turtle1/cmd_vel geometry_msgs/msg/Twist "{linear: {x: 1.0}}"
```

The message value is YAML. The field names come from the message type definition — `ros2 interface show geometry_msgs/msg/Twist` lists them.
