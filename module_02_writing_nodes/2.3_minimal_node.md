## Step 1 — Write the Node

By convention, all Python node scripts for a package go inside the inner Python module directory:

```
src/<package_name>/<package_name>/
```

For this package that is:

```
~/tutorial_ws/src/turtlesim_controller/turtlesim_controller/
```

This is the same directory that already contains `__init__.py`. Create a new file there called `minimal_node.py` and paste in the following:

```
import rclpy
from rclpy.node import Node


class MinimalNode(Node):
    def __init__(self):
        super().__init__('minimal_node')
        self.get_logger().info('MinimalNode started.')


def main(args=None):
    rclpy.init(args=args)
    node = MinimalNode()
    try:
        rclpy.spin(node)
    except KeyboardInterrupt:
        pass
    finally:
        node.destroy_node()
        if rclpy.ok():
            rclpy.shutdown()


if __name__ == '__main__':
    main()
```

### What each part does

| Line | Purpose |
|---|---|
| `import rclpy` | Imports the ROS2 Python client library |
| `from rclpy.node import Node` | Imports the base class for all ROS2 nodes |
| `class MinimalNode(Node)` | Your node is a Python class that inherits from `Node` |
| `super().__init__('minimal_node')` | Registers the node with ROS2 under the name `minimal_node` |
| `self.get_logger().info(...)` | Logs a message at INFO level — visible in the terminal |
| `rclpy.init(args=args)` | Connects to the ROS2 middleware; must be called before anything else |
| `rclpy.spin(node)` | Runs the event loop — explained in detail below |
| `node.destroy_node()` | Releases ROS2 resources held by this node |
| `rclpy.shutdown()` | Disconnects from the middleware |

According to this script, the node will print a logging statement "MinimalNode started" once in the beginning, and remain idle for the rest of its process.
---

### The event loop — why spin instead of while True

Every useful node needs to run continuously: waiting for incoming messages, firing timers, responding to service calls. The naive way to do this is an explicit loop:

```python
while True:
    # check for new messages
    # fire any due timers
    # handle service calls
```

`rclpy.spin(node)` does exactly this, but correctly. It processes the ROS2 event queue in a loop — checking for any pending callbacks (timer ticks, incoming topic messages, incoming service requests) and calling them one at a time. When there is nothing to do, spin yields the CPU rather than busy-waiting.

You never write the loop yourself. Instead, you register **callbacks** — functions that spin will call when a particular event occurs. In 2.4 you will register a timer callback; in 2.5 and 2.6 you will register publisher and subscriber callbacks. The loop is always implied; spin runs it.

---

### Graceful exit

When the user presses `Ctrl+C`, Python raises a `KeyboardInterrupt`. Without the `try/except/finally` block, the node would crash without cleaning up — ROS2 might not release the node's resources cleanly and could log spurious errors.

The pattern used here matches the existing nodes in this repository:

```python
try:
    rclpy.spin(node)
except KeyboardInterrupt:
    pass          # Ctrl+C is expected — not an error
finally:
    node.destroy_node()       # always release node resources
    if rclpy.ok():
        rclpy.shutdown()      # disconnect from middleware if still connected
```

`finally` runs whether the exit was clean (`Ctrl+C`) or due to an exception, so cleanup always happens.

## Step 2 — Register the Entry Point

For `ros2 run` to find your node, you need to tell colcon that `minimal_node.py` contains a runnable executable. This is done in `setup.py`.

Open `~/tutorial_ws/src/turtlesim_controller/setup.py`. Find the `entry_points` section near the bottom:

```python
entry_points={
    'console_scripts': [
        # executables go here
    ],
},
```

Each line you add to `console_scripts` creates one `ros2 run` command. The format is:

```
'<executable_name> = <package_name>.<module_name>:<function_name>'
```

Add one line so the section looks like this:

```python
entry_points={
    'console_scripts': [
        'minimal_node = turtlesim_controller.minimal_node:main',
    ],
},
```

This tells `ros2 run turtlesim_controller minimal_node` to call the `main()` function inside `turtlesim_controller/minimal_node.py`. Save the file.

## Alternative Design — Composition over Inheritance

The implementation above uses **inheritance**: `MinimalNode` inherits from `Node`, so it *is* a Node. This is the pattern used throughout the ROS2 Python tutorials and is what you will use going forward.

There is an alternative called **composition**: instead of inheriting from `Node`, your class *holds* a `Node` as an attribute. The class *has* a node rather than *being* one.

Here is the same minimal node written with composition:

```
import rclpy
from rclpy.node import Node


class MinimalNode:                          # no base class
    def __init__(self):
        self.node = Node('minimal_node')    # Node is an attribute, not a parent
        self.node.get_logger().info('MinimalNode started.')


def main(args=None):
    rclpy.init(args=args)
    minimal = MinimalNode()
    try:
        rclpy.spin(minimal.node)            # pass the Node attribute to spin
    except KeyboardInterrupt:
        pass
    finally:
        minimal.node.destroy_node()
        if rclpy.ok():
            rclpy.shutdown()


if __name__ == '__main__':
    main()
```

### Comparing the two approaches

| | Inheritance (`class MinimalNode(Node)`) | Composition (`class MinimalNode`) |
|---|---|---|
| Relationship | MinimalNode **is** a Node | MinimalNode **has** a Node |
| Logger access | `self.get_logger()` | `self.node.get_logger()` |
| Pass to spin | `rclpy.spin(self)` | `rclpy.spin(self.node)` |
| Cleanup | `self.destroy_node()` | `self.node.destroy_node()` |

Both produce an identical running node from ROS2's perspective. The difference is in the software design.

**Why inheritance is common in ROS2:** The `Node` base class exposes dozens of methods — `create_publisher`, `create_subscription`, `create_timer`, `get_logger`, `declare_parameter`, and more. Inheriting from it makes all of these available directly as `self.method()`, which keeps node code concise and matches every official ROS2 tutorial.

**Why composition is worth knowing:** In larger systems, inheritance can become limiting. If your class needs to serve multiple roles, deep inheritance chains become hard to maintain. Composition lets you swap the underlying `Node` freely, combine behaviours without coupling, and test your logic independently of ROS2. This principle — preferring composition over inheritance — is a widely recognised guideline in object-oriented software design.

This tutorial uses inheritance going forward. If you encounter real-world ROS2 codebases that use composition, you now know why.

## Summary

| Concept | Key point |
|---|---|
| `package.xml` `<depend>` | Declares a package dependency; add one line per new dependency |
| `setup.py` `console_scripts` | Maps `ros2 run` executable names to Python `main()` functions |
| Entry point format | `'name = package.module:function'` |
| `rclpy.init()` | Connects to ROS2 middleware — call once at the start of `main()` |
| `rclpy.spin(node)` | Runs the event loop; callbacks fire here; blocks until `Ctrl+C` |
| `rclpy.shutdown()` | Cleans up after the event loop exits |
| Rebuild after editing | Required any time you edit `setup.py` or add new node files |
