## A Subscriber That Never Fires

Here is a scenario you will likely encounter at some point: you write a subscriber to an existing topic. The node starts without errors. `rqt_graph` shows a clean arrow from the publisher to your subscriber. Yet the callback never fires â€” not once.

The cause is a **QoS mismatch**. The publisher and subscriber agreed on a topic name, but they disagree on the delivery policy. When that disagreement breaks the compatibility rules, the DDS transport silently drops the connection and delivers no data.

This section explains why that happens and how to find and fix it.

---

## Worked Example â€” Diagnosing a Silent Subscriber

### Step 1 â€” Run turtlesim

**Terminal 1:**

```bash
ros2 run turtlesim turtlesim_node
```

### Step 2 â€” Write `pose_listener.py`

Create the file:

```
~/tutorial_ws/src/turtlesim_controller/turtlesim_controller/pose_listener.py
```

```python
import rclpy
from rclpy.node import Node
from rclpy.qos import QoSProfile, ReliabilityPolicy
from turtlesim.msg import Pose


class PoseListener(Node):
    def __init__(self):
        super().__init__('pose_listener')
        qos = QoSProfile(depth=10, reliability=ReliabilityPolicy.RELIABLE)
        self.create_subscription(Pose, '/turtle1/pose', self.pose_callback, qos)
        self.get_logger().info('Listening for pose...')

    def pose_callback(self, msg):
        self.get_logger().info(f'x={msg.x:.2f}  y={msg.y:.2f}  theta={msg.theta:.2f}')


def main(args=None):
    rclpy.init(args=args)
    node = PoseListener()
    try:
        rclpy.spin(node)
    except KeyboardInterrupt:
        pass
    finally:
        node.destroy_node()
        if rclpy.ok():
            rclpy.shutdown()


if __name__ == '__main__':
    main()
```

Register the entry point in `setup.py`:



Output (abbreviated):

```
...

  Reliability: BEST_EFFORT

  Durability: VOLATILE
  ...

...

  Reliability: RELIABLE
  History (Depth): KEEP_LAST (10)
  Durability: VOLATILE
  ...
```

The mismatch is on the `Reliability` line:

- Publisher: `BEST_EFFORT`
- Subscriber: `RELIABLE`

The subscriber is demanding guaranteed delivery from a publisher that only offers best-effort. That combination is incompatible â€” no data flows.

### Step 6 â€” Fix It

Replace the explicit `QoSProfile` with `QoSPresetProfiles.SENSOR_DATA`, which uses `BEST_EFFORT` reliability â€” matching what turtlesim publishes:

```python
from rclpy.qos import QoSPresetProfiles
```

```python
self.create_subscription(
    Pose, '/turtle1/pose', self.pose_callback,
    QoSPresetProfiles.SENSOR_DATA.value
)
```

The complete fixed file:

```python

```

## QoS Policies

Now that you have seen the problem and the fix, here is the underlying model.

**Quality of Service (QoS)** is a set of policies that control how messages are delivered between a publisher and a subscriber. Both sides declare their policies; DDS enforces compatibility before connecting them.

### The Four Key Policies

**Reliability** â€” what happens if a message is lost in transit:

| Value | Meaning |
|---|---|
| `RELIABLE` | Every message is delivered; retransmit on loss |
| `BEST_EFFORT` | Send once; dropped messages are gone |

**Durability** â€” what late-joining subscribers receive:

| Value | Meaning |
|---|---|
| `VOLATILE` | Late joiners miss messages published before they connected |
| `TRANSIENT_LOCAL` | Publisher caches recent messages; late joiners receive them on connect |

**History** â€” how many messages the queue holds:

| Value | Meaning |
|---|---|
| `KEEP_LAST(N)` | Keep the N most recent messages |
| `KEEP_ALL` | Keep every message (bounded by system memory) |

**Depth** â€” the queue size when using `KEEP_LAST`. Passing `10` as the QoS argument to `create_subscription` or `create_publisher` is shorthand for `QoSProfile(depth=10)`, which uses `RELIABLE`, `VOLATILE`, and `KEEP_LAST(10)` â€” the system defaults.

### The Compatibility Rule

A connection is only established when the subscriber does not request *more* than the publisher offers:

| Publisher | Subscriber | Compatible? |
|---|---|---|
| `RELIABLE` | `RELIABLE` | âś“ |
| `RELIABLE` | `BEST_EFFORT` | âś“ (subscriber accepts less) |
| `BEST_EFFORT` | `BEST_EFFORT` | âś“ |
| `BEST_EFFORT` | `RELIABLE` | âś— subscriber demands guarantees the publisher can't give |
| `TRANSIENT_LOCAL` | `VOLATILE` | âś“ |
| `TRANSIENT_LOCAL` | `TRANSIENT_LOCAL` | âś“ |
| `VOLATILE` | `VOLATILE` | âś“ |
| `VOLATILE` | `TRANSIENT_LOCAL` | âś— subscriber demands history the publisher doesn't keep |

When incompatible, no messages flow and no error is raised â€” the failure is **silent**.

---

## Common Presets

`QoSPresetProfiles` bundles the policies you use most often:

| Preset | Reliability | Durability | Depth | Use for |
|---|---|---|---|---|
| `SYSTEM_DEFAULT` | `RELIABLE` | `VOLATILE` | 10 | Commands, services, most inter-node traffic |
| `SENSOR_DATA` | `BEST_EFFORT` | `VOLATILE` | 5 | High-rate sensor streams (pose, scan, image) |

Use `QoSPresetProfiles.SENSOR_DATA.value` when subscribing to any topic published by a sensor driver or simulator at high rate. Use `SYSTEM_DEFAULT` (or just the integer depth) for everything else.

The publisher's profile shows `Reliability: BEST_EFFORT`.

1. **`BEST_EFFORT`** â€” turtlesim publishes pose at ~62 Hz and makes no delivery guarantees.

## Summary

| Concept | Key point |
|---|---|
| `RELIABLE` vs `BEST_EFFORT` | RELIABLE retransmits on loss; BEST_EFFORT sends once and moves on |
| `VOLATILE` vs `TRANSIENT_LOCAL` | VOLATILE: late joiners miss past messages; TRANSIENT_LOCAL: publisher caches them |
| Compatibility rule | Subscriber cannot demand more than the publisher offers â€” mismatch silences the connection |
| Why `rqt_graph` misleads | Draws arrows based on topic name match; does not check QoS compatibility |
| `ros2 topic info --verbose` | Shows publisher and subscriber QoS profiles side by side â€” the right tool to find a mismatch |
| `QoSPresetProfiles.SENSOR_DATA` | `BEST_EFFORT`, `VOLATILE`, depth 5 â€” correct for high-rate sensor and simulator topics |
| `QoSPresetProfiles.SYSTEM_DEFAULT` | `RELIABLE`, `VOLATILE`, depth 10 â€” the default; correct for command topics |
| Passing `10` as QoS | Shorthand for `QoSProfile(depth=10)` â€” uses `RELIABLE` by default |
