## Creating a Publisher

A publisher lets your node send messages to a topic. You create one in `__init__` and call `.publish()` in a callback:

```python
# In __init__:
self.pub = self.create_publisher(String, 'chatter', 10)

# In a callback:
msg = String()
msg.data = 'Hello, world!'
self.pub.publish(msg)
```

`self.create_publisher` takes three arguments:

| Argument | Example | Meaning |
|---|---|---|
| Message type | `String` | The Python class for this message type |
| Topic name | `'chatter'` | The topic to publish on |
| QoS depth | `10` | History depth — see below |

### QoS depth

The third argument is the **history depth**: how many outgoing messages to keep in the queue if a subscriber is too slow to consume them. Once the queue reaches the limit, the oldest messages are dropped to make room for new ones.

For a 1 Hz publisher, a depth of `10` means up to 10 seconds of messages could be buffered — far more than needed. In practice, `10` is the conventional starting value for the vast majority of topics. You would only raise it if your publisher is very fast and your subscriber is known to be slow.

QoS has other dimensions beyond history depth (reliability, durability, liveliness) that control how messages behave across unreliable networks and late-joining subscribers. Those are outside the scope of this notebook; the integer shorthand sets only the history depth and accepts sensible defaults for everything else.

## Message Types and Imports

Every ROS2 message type follows the same naming convention:

```
<package>/msg/<TypeName>
```

The Python import maps directly from this name — replace `/msg/` with `.msg.` and import the class:

| Full message name | Python import |
|---|---|
| `std_msgs/msg/String` | `from std_msgs.msg import String` |
| `geometry_msgs/msg/Twist` | `from geometry_msgs.msg import Twist` |
| `turtlesim/msg/Pose` | `from turtlesim.msg import Pose` |

You will see the full name everywhere: in `ros2 topic info`, in `ros2 topic echo` headers, and in documentation. Once you know the pattern, you can always derive the import.

### Discovering message fields

To see what fields a message type has, use `ros2 interface show`:

```bash
ros2 interface show std_msgs/msg/String
```

Output:
```
string data
```

`String` has one field: `data`, of type `string`. To instantiate and fill it in Python:

```python
msg = String()
msg.data = 'Hello, world!'
```

Try the same command on `geometry_msgs/msg/Twist` or any other type you encounter — the output tells you exactly which fields exist and what types they are.

### Imports and dependencies

The `<package>` part of the message name must be declared as a dependency in `package.xml`. If you write `from std_msgs.msg import String` but `std_msgs` is not listed, the build will succeed on your machine by accident — but fail elsewhere. The rule is simple: if you import from a package, declare it. How packages get installed on a given machine is covered in a later submodule.

## Step 1 — Declare the New Dependency

`turtlesim_controller` was created with `rclpy` and `geometry_msgs`. The `String` message type lives in `std_msgs`, which has not been declared yet.

`package.xml` is where a ROS2 package lists everything it depends on. Colcon reads this file during the build to verify that each listed package is present and to wire up the correct import paths. A missing entry may work by accident on your machine if the package happens to already be installed for some other reason — but it will fail when the package is built from scratch on a clean system. Declaring dependencies explicitly is what makes a package reliably buildable anywhere.

Open `~/tutorial_ws/src/turtlesim_controller/package.xml`. Find the existing `<depend>` lines and add `std_msgs` immediately after them:

```xml
  <depend>rclpy</depend>
  <depend>geometry_msgs</depend>
  <depend>std_msgs</depend>    <!-- add this line -->

  <test_depend>ament_copyright</test_depend>
```

## Step 2 — Write the Node

Create `hello_publisher.py` inside `~/tutorial_ws/src/turtlesim_controller/turtlesim_controller/` and paste in the following:

```
import rclpy
from rclpy.node import Node
from std_msgs.msg import String


class HelloPublisher(Node):
    def __init__(self):
        super().__init__('hello_publisher')
        self.pub = self.create_publisher(String, 'chatter', 10)
        self.create_timer(1.0, self.timer_callback)

    def timer_callback(self):
        msg = String()
        msg.data = 'Hello, world!'
        self.pub.publish(msg)


def main(args=None):
    rclpy.init(args=args)
    node = HelloPublisher()
    try:
        rclpy.spin(node)
    except KeyboardInterrupt:
        pass
    finally:
        node.destroy_node()
        if rclpy.ok():
            rclpy.shutdown()


if __name__ == '__main__':
    main()
```

### What is new compared to 2.4

| Addition | What it does |
|---|---|
| `from std_msgs.msg import String` | Imports the standard string message type |
| `self.create_publisher(String, 'chatter', 10)` | Registers a publisher on the `/chatter` topic |
| `msg = String()` | Creates a new message; `data` defaults to an empty string |
| `msg.data = 'Hello, world!'` | Sets the one field that `String` has |
| `self.pub.publish(msg)` | Sends the message |

The `String` message type has a single field: `data`. The timer fires at 1 Hz, so the node publishes one message per second.

`/chatter` is the canonical topic name used in ROS hello-world examples.

Press `Ctrl+C` to stop the echo. The publisher keeps running in Terminal 1.

You can also confirm the topic exists and has a publisher:

```
ros2 topic info /chatter
```

Expected output:

```
Type: std_msgs/msg/String
Publisher count: 1
Subscription count: 0
```

`Subscription count: 0` — `ros2 topic echo` subscribes temporarily and disconnects when you press `Ctrl+C`. The publisher does not know or care; it keeps publishing regardless. This is pub/sub decoupling: **publishers and subscribers are unaware of each other's existence**.

Press `Ctrl+C` in Terminal 1 to stop `hello_publisher`.

## REP 103 — Standard Units

Before writing a velocity publisher, you need to know the units ROS2 expects. [REP 103](https://www.ros.org/reps/rep-0103.html) defines the conventions every ROS2 node is expected to follow:

| Quantity | Unit |
|---|---|
| Length / position | metres (m) |
| Angle | radians (rad) |
| Linear velocity | metres per second (m/s) |
| Angular velocity | radians per second (rad/s) |

For coordinate axes on a ground robot, the convention is:
- **x** — forward
- **y** — left
- **z** — up

When you publish `linear.x = 1.0` in a velocity command, that value is **1.0 m/s forward**. Any node receiving the command — a real motor driver or a simulator — interprets it the same way because every publisher is expected to use the same units.

---

## REP 105 — Coordinate Frames

[REP 105](https://www.ros.org/reps/rep-0105.html) defines the standard named coordinate frames used in mobile robotics:

| Frame | Meaning |
|---|---|
| `base_link` | The robot body — attached to the physical robot |
| `base_footprint` | Ground-projected centre of the robot |
| `odom` | Continuous odometry frame — drifts over time, never jumps |
| `map` | Globally consistent frame — may jump when loop closure occurs |

These names appear as the `frame_id` field inside message headers. A `Pose` message describing where the robot is will carry `header.frame_id = 'odom'` or `'map'` so that any subscriber knows which coordinate frame the pose is expressed in.

The `Twist` message used for velocity commands has no header — velocity is always expressed relative to the robot body, so no frame annotation is needed. You will encounter `frame_id` in later notebooks when working with poses and sensor data.

## Exercise — Drive the Turtle in a Circle

Write a node called `circle_publisher` that drives a turtlesim turtle in a circle by publishing `geometry_msgs/msg/Twist` messages to `/turtle1/cmd_vel`. Aim for a circle with a **radius of approximately 1.0 m** and a **period of approximately 6 seconds**.

> **Hint:** `geometry_msgs` is already declared as a dependency — no `package.xml` change needed.

---
---

## Answer Key

**Deriving the values:**

```
angular.z  =  2π / 6  ≈  1.047 rad/s
linear.x   =  radius × angular.z  =  1.0 × 1.047  ≈  1.047 m/s
```

```
import rclpy
from rclpy.node import Node
from geometry_msgs.msg import Twist


class CirclePublisher(Node):
    def __init__(self):
        super().__init__('circle_publisher')
        self.pub = self.create_publisher(Twist, '/turtle1/cmd_vel', 10)
        self.create_timer(0.1, self.timer_callback)

    def timer_callback(self):
        msg = Twist()
        msg.linear.x = 1.047   # 1.0 m radius × 1.047 rad/s
        msg.angular.z = 1.047  # 2π / 6 s
        self.pub.publish(msg)


def main(args=None):
    rclpy.init(args=args)
    node = CirclePublisher()
    try:
        rclpy.spin(node)
    except KeyboardInterrupt:
        pass
    finally:
        node.destroy_node()
        if rclpy.ok():
            rclpy.shutdown()


if __name__ == '__main__':
    main()
```

**Entry point to add in setup.py:**

```python
'circle_publisher = turtlesim_controller.circle_publisher:main',
```

**Launch sequence:**

```
ros2 run turtlesim turtlesim_node
```

```
ros2 run turtlesim_controller circle_publisher
```

## Summary

| Concept | Key point |
|---|---|
| `create_publisher(Type, topic, depth)` | Registers a publisher; returns a handle to call `.publish()` on |
| `msg = Type()` | Creates a message with all fields at their zero/empty default |
| QoS depth 10 | Standard queue size |
| `ros2 topic echo <topic>` | Prints incoming messages to the terminal |
| REP 103 | All velocities in m/s and rad/s; x forward, y left, z up |
| REP 105 | Standard frame names: `base_link`, `odom`, `map`; carried in `header.frame_id` |
| Pub/sub decoupling | Publishers and subscribers are unaware of each other; the topic is the only shared contract |
