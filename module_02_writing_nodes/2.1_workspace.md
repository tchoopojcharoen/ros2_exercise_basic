# 2.1 — Creating a ROS2 Workspace

## Learning Objectives

By the end of this section, you will be able to:
- Explain what a ROS2 workspace is and why it exists
- Create a workspace with the correct directory structure
- Build an empty workspace with `colcon`
- Source the workspace overlay so your packages are discoverable

---

## Before You Begin

This notebook assumes nothing is running. You only need a single terminal.

Make sure the system ROS2 underlay is sourced in your terminal before continuing:

```bash
source /opt/ros/humble/setup.bash
```

---

## What Is a ROS2 Workspace?

A **workspace** is a directory that contains everything you write and build for a ROS2 project. It gives `colcon` (the ROS2 build tool) a known structure to work with:

```
tutorial_ws/       ← workspace root
├── src/           ← your source packages go here
├── build/         ← intermediate build files (created by colcon)
├── install/       ← compiled outputs and setup files (created by colcon)
└── log/           ← build logs (created by colcon)
```

You only create `src/` by hand. `colcon` creates the other three directories the first time you build.

All the ROS2 nodes, message types, and libraries you write will live inside `src/`, organised into **packages** (covered in notebook 2.2).

A workspace can be placed anywhere on the filesystem — your home directory, a project folder, an external drive. The only hard rule is that **workspaces must never be nested inside one another**. Placing `~/tutorial_ws` inside another workspace's `src/` (for example, `~/ros2_ws/src/tutorial_ws`) confuses colcon and causes unpredictable build failures.

---

## Underlay and Overlay

ROS2 uses a layered system of sourced environments:

| Layer | What it is | Where it lives |
|---|---|---|
| **Underlay** | System-wide ROS2 installation | `/opt/ros/humble/` |
| **Overlay** | Your workspace | `~/tutorial_ws/install/` |

Sourcing the underlay makes `ros2`, `colcon`, and all built-in packages available. Sourcing your overlay *on top of* that adds your own packages to the same environment.

The order matters: always source the underlay first, then the overlay. If `/opt/ros/humble/setup.bash` is already in your `~/.bashrc`, the underlay is sourced automatically every time you open a terminal.

You must source the overlay (`source ~/tutorial_ws/install/setup.bash`) in **every new terminal** that needs your packages. The overlay does not persist across terminals unless you add it to `~/.bashrc`.

---

## Step 1 — Create the Workspace

Create the workspace root and its `src/` subdirectory in one command:

```
mkdir -p ~/tutorial_ws/src
```

The `-p` flag creates parent directories as needed and does not fail if they already exist.

Your workspace directory is now at `~/tutorial_ws/`. It contains only `src/` — everything else is created by `colcon` in the next step.

---

## Step 2 — Build the Empty Workspace

Navigate into the workspace and run `colcon build`:

```
cd ~/tutorial_ws && colcon build
```

Colcon scans `src/` for packages, finds none, and exits cleanly. You should see:

```
Summary: 0 packages finished [<0.1s]
```

This is expected — you have not created any packages yet. Even with an empty `src/`, colcon still creates the `build/`, `install/`, and `log/` directories, which you need before you can source the workspace.

---

## Step 3 — Inspect the Created Directories

Confirm that colcon created the expected structure:

```
ls ~/tutorial_ws
```

Expected output:

```
build  install  log  src
```

| Directory | Purpose |
|---|---|
| `src/` | Your packages live here — this is where you write code |
| `build/` | Intermediate build artifacts; managed by colcon, not edited directly |
| `install/` | Final compiled outputs and `setup.bash` — what you source |
| `log/` | Build logs; useful for diagnosing failed builds |

You will never need to edit anything inside `build/`, `install/`, or `log/` by hand.

---

## Step 4 — Source the Workspace Overlay

Make your workspace visible to ROS2 by sourcing its overlay:

```
source ~/tutorial_ws/install/setup.bash
```

This command produces no output on success. It extends your shell's `PATH` and `PYTHONPATH` so that `ros2 run`, `ros2 pkg list`, and Python imports can find packages installed in this workspace.

> **Every new terminal needs this.** Sourcing only affects the shell it runs in. Open a new terminal and the workspace is invisible again until you source it there too.
>
> **After every build, re-source.** When you build a new or updated package, the overlay changes. Source it again in any terminal that was already open before the build.

---

## Step 5 — Add the Overlay to ~/.bashrc

From notebook 2.3 onward you will run nodes across multiple terminals simultaneously. Rather than sourcing the workspace in each terminal by hand, add the source command to `~/.bashrc` so every new terminal picks it up automatically:

```
echo "source ~/tutorial_ws/install/setup.bash" >> ~/.bashrc
```

Every terminal you open from this point on will automatically have the workspace sourced. The terminal you have open right now was sourced manually in Step 4 and needs no further action.

---

## Summary

| Concept | Key point |
|---|---|
| Workspace | A directory with `src/`, `build/`, `install/`, `log/` |
| `src/` | You create this; packages go inside it |
| `colcon build` | Compiles packages in `src/`; creates `build/`, `install/`, `log/` |
| Underlay | System ROS2 at `/opt/ros/humble/` |
| Overlay | Your workspace at `~/tutorial_ws/install/` |
| Sourcing | `~/.bashrc` handles new terminals; re-source manually after each build |
