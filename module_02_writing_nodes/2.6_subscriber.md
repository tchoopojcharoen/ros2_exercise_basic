## Creating a Subscriber

A subscriber lets your node receive messages from a topic. You create one in `__init__` and handle each incoming message in a callback:

```python
# In __init__:
self.create_subscription(Pose, '/turtle1/pose', self.pose_callback, 10)

# The callback — called by spin whenever a message arrives:
def pose_callback(self, msg):
    self.get_logger().info(f'x={msg.x:.2f}')
```

`self.create_subscription` takes four arguments:

| Argument | Example | Meaning |
|---|---|---|
| Message type | `Pose` | The Python class for messages on this topic |
| Topic name | `'/turtle1/pose'` | The topic to subscribe to |
| Callback | `self.pose_callback` | Method to call when a message arrives |
| QoS depth | `10` | History depth — same meaning as for publishers |

### Callback signature

Subscriber callbacks take one argument beyond `self`:

```python
def pose_callback(self, msg):   # msg is the received message object
```

This is different from a timer callback, which takes no arguments:

```python
def timer_callback(self):       # no msg — timer fires on schedule, not on data
```

The `msg` object has the same fields as the message type. Access them with dot notation. The callback fires at whatever rate the publisher is sending — spin dispatches each arriving message automatically.

## Step 2 — Write the Node

Create `pose_reporter.py` inside `~/tutorial_ws/src/turtlesim_controller/turtlesim_controller/` and paste in the following:

Here is what `pose_reporter` will do:

1. It subscribes to `/turtle1/pose`. Turtlesim publishes that topic at approximately 60 Hz.
2. Each time a pose message arrives on the topic, spin fires `pose_callback` and passes the message in as `msg`.
3. The callback stores the message, computes the straight-line distance from the turtle to the zone centre (5.44, 5.44), and immediately publishes `True` or `False` to `/in_zone`.

The key point: **the callback executes because a message arrived, not because a timer fired.** Spin watches the topic; when a new message is delivered, spin calls your method. Because the callback publishes on every incoming message, the `/in_zone` publish rate matches the subscribe rate (~60 Hz). The exercise at the end of this notebook will decouple those two rates.

```
import math

import rclpy
from rclpy.node import Node
from turtlesim.msg import Pose
from std_msgs.msg import Bool

ZONE_CENTER_X = 5.44
ZONE_CENTER_Y = 5.44
ZONE_RADIUS = 5.0


class PoseReporter(Node):
    def __init__(self):
        super().__init__('pose_reporter')
        self.latest_pose = None
        self.pub = self.create_publisher(Bool, 'in_zone', 10)
        self.create_subscription(Pose, '/turtle1/pose', self.pose_callback, 10)

    def pose_callback(self, msg):
        self.latest_pose = msg
        d = math.sqrt((msg.x - ZONE_CENTER_X) ** 2 + (msg.y - ZONE_CENTER_Y) ** 2)
        result = Bool()
        result.data = d <= ZONE_RADIUS
        self.pub.publish(result)


def main(args=None):
    rclpy.init(args=args)
    node = PoseReporter()
    try:
        rclpy.spin(node)
    except KeyboardInterrupt:
        pass
    finally:
        node.destroy_node()
        if rclpy.ok():
            rclpy.shutdown()


if __name__ == '__main__':
    main()
```

### What is new compared to 2.5

| Addition | What it does |
|---|---|
| `from std_msgs.msg import Bool` | Imports the boolean message type |
| `ZONE_CENTER_X/Y`, `ZONE_RADIUS` | Module-level constants — keeps magic numbers out of methods |
| `self.latest_pose = None` | Class attribute initialised in `__init__`; updated each callback |
| `self.create_subscription(...)` | Registers `pose_callback` on `/turtle1/pose` |
| `def pose_callback(self, msg)` | Receives the message as `msg`; fields accessed with dot notation |
| `self.latest_pose = msg` | Stores the full message object on the node for later use |
| `d = math.sqrt(...)` | Euclidean distance from the turtle to the zone centre |
| `result.data = d <= ZONE_RADIUS` | `True` if inside the zone, `False` if outside |
| `self.pub.publish(result)` | Publisher called from inside the subscriber callback |

A callback is just a plain Python method — it can read and write class attributes, call other methods, and use publishers. There is nothing special about calling `publish` from inside a subscriber callback.

`self.latest_pose` starts as `None`. Checking `if self.latest_pose is None` before using it is the standard guard pattern for attributes that depend on a message arriving. You will see this again in 2.7.

## Answer Key

```
import math

import rclpy
from rclpy.node import Node
from turtlesim.msg import Pose
from std_msgs.msg import Bool

ZONE_CENTER_X = 5.44
ZONE_CENTER_Y = 5.44
ZONE_RADIUS = 5.0

class PoseReporter(Node):
    def __init__(self):
        super().__init__('pose_reporter')
        self.latest_pose = None
        self.pub = self.create_publisher(Bool, 'in_zone', 10)
        self.create_subscription(Pose, '/turtle1/pose', self.pose_callback, 10)
        self.create_timer(0.5, self.report_callback)

    def pose_callback(self, msg):
        self.latest_pose = msg

    def report_callback(self):
        if self.latest_pose is None:
            return
        d = math.sqrt(
            (self.latest_pose.x - ZONE_CENTER_X) ** 2
            + (self.latest_pose.y - ZONE_CENTER_Y) ** 2
        )
        result = Bool()
        result.data = d <= ZONE_RADIUS
        self.pub.publish(result)


def main(args=None):
    rclpy.init(args=args)
    node = PoseReporter()
    try:
        rclpy.spin(node)
    except KeyboardInterrupt:
        pass
    finally:
        node.destroy_node()
        if rclpy.ok():
            rclpy.shutdown()


if __name__ == '__main__':
    main()
```

The key change is that `pose_callback` now does only one thing: store the message. All computation and publishing moves to `report_callback`, which a 0.5 s timer fires independently of how fast messages arrive.

The `None` guard in `report_callback` is essential. If the timer fires before turtlesim has published a single pose, `self.latest_pose` is still `None` and accessing `.x` would crash the node. The guard makes the timer a no-op until real data arrives.

This subscriber-stores / timer-publishes split is the standard pattern whenever a node subscribes to one rate but needs to act at a different rate. You will use exactly the same structure in 2.7: two subscriber callbacks store pose and goal, and a separate 10 Hz timer reads both and publishes the velocity command.

## Summary

| Concept | Key point |
|---|---|
| `create_subscription(Type, topic, callback, depth)` | Registers a callback to fire on every incoming message |
| Callback fires on message arrival | Not on a schedule — spin dispatches it when a message lands on the topic |
| Callback signature | `def callback(self, msg):` — `msg` carries the received data |
| Timer vs subscriber callback | Timer: `(self)` — no data. Subscriber: `(self, msg)` — receives the message |
| Class attribute for state | Initialise to `None` in `__init__`; update in the callback |
| `None` guard | Check `if self.attr is None: return` before using a value that depends on a message arriving |
| Subscriber-stores / timer-publishes | Store latest data in subscriber callback; read and act in a separate timer callback to decouple rates |
